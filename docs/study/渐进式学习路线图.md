# AI 竞品分析 Agent 协作系统 — 渐进式学习路线图

## 项目全景

### 一句话概括

这是一个 **Spring Boot + LangChain4j** 的多 Agent 协作系统，模拟一个数字调研小组，**全自动完成**从网上采集竞品信息 → 结构化抽取 → 对比分析 → 撰写报告 → 引用核查 → 质量评审的全链路工作。

### 技术栈速览

| 层 | 技术 |
|---|---|
| 后端框架 | Spring Boot 3.3.5, Java 17 |
| LLM 集成 | LangChain4j 0.35.0（支持 OpenAI / 百炼 / DeepSeek / Ollama 等） |
| 数据库 | PostgreSQL + Redis + H2（开发环境） |
| 消息队列 | RocketMQ（事件驱动架构） |
| 网页抓取 | Playwright + Jsoup |
| 前端 | React 18 + TypeScript + Ant Design 5 + Vite |
| 项目规模 | 后端 **463 个 Java 文件**，前端 ~40 个 TSX 文件 |

### 核心业务流水线（DAG）

```
采集 Agent ──→ 抽取 Agent ──→ 分析 Agent ──→ 撰写 Agent ──→ 引用核查 Agent ──→ 质量评审 Agent
 (Collector)    (Extractor)    (Analyzer)     (Writer)        (Citation)         (Reviewer)
     │                                                              │                  │
     └──── 如果评审不通过，自动生成补充采集 / 重写分支 ←────────────────┘
```

## 🗄️ 数据库设计（26 张表）

> 所有表之间**不使用 JPA 外键注解**（`@ManyToOne`、`@OneToMany` 等），表间引用全部通过应用层字段（如 `task_id`、`node_id`）手动管理。主键统一使用 `BIGSERIAL` 自增 ID。

### 一、核心业务表（7 张）

这些表构成了竞品分析业务的主链路。

#### 1. `analysis_task` — 分析任务

**职责**：存储每一次竞品分析任务的元信息。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `BIGSERIAL PK` | 主键 |
| `task_name` | `VARCHAR(200)` | 任务名称 |
| `subject_product` | `VARCHAR(200)` | 目标产品名称 |
| `competitor_names` | `TEXT` (JSON 数组) | 竞品名称列表，如 `["Notion AI","Glean"]` |
| `competitor_urls` | `TEXT` (JSON 数组) | 竞品官网 URL 列表 |
| `analysis_dimensions` | `TEXT` (JSON 数组) | 分析维度，如 `["产品功能","定价"]` |
| `source_scope` | `TEXT` (JSON 数组) | 信息源范围 |
| `report_language` | `VARCHAR(20)` | 报告语言，默认 `"中文"` |
| `report_template` | `VARCHAR(50)` | 报告模板，默认 `"标准版"` |
| `status` | `VARCHAR(20)` | 任务状态：PENDING / RUNNING / SUCCESS / FAILED / STOPPED |
| `schema_id` | `BIGINT` | 关联的分析模板 ID |
| `task_quota_reserved` | `BOOLEAN` | 配额是否已预留 |
| `current_plan_version_id` | `BIGINT` | 当前生效的 TaskPlan ID |
| `current_plan_version` | `INTEGER` | 当前计划版本号 |
| `error_message` | `TEXT` | 错误信息 |
| `created_at` / `updated_at` / `started_at` / `completed_at` | `TIMESTAMP` | 时间戳 |

**关联**：
- `schema_id` → `analysis_schema.id`
- `id` ← `task_node.task_id`（一个任务有多个节点）
- `id` ← `report.task_id`（一个任务产生一份报告）

#### 2. `task_node` — DAG 任务节点

**职责**：DAG 工作流中的每个执行节点——这是工作流引擎最核心的表。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `BIGSERIAL PK` | 主键 |
| `task_id` | `BIGINT` | 所属任务 ID |

| `display_name` | `VARCHAR(100)` | 节点显示名，如 `"采集公开信息"` |

| `node_config` | `TEXT` (JSON) | 节点专属配置（如 Collector 的搜索策略） |
| `node_notes` | `TEXT` | 节点备注 |
| `allow_failed_dependency` | `BOOLEAN` | 是否允许上游节点失败后继续 |
| `required` | `BOOLEAN` | 是否为必须节点 |
| `retryable` | `BOOLEAN` | 是否可重试 |
| `max_retries` | `INT` | 最大重试次数，默认 3 |
| `retry_count` | `INT` | 已重试次数 |
| `failure_category` | `VARCHAR(50)` | 失败分类（见下文枚举） |
| `status` | `VARCHAR(32)` | 节点状态（见下文枚举） |
| `control_state` | `VARCHAR(30)` | 控制状态：NONE / PAUSE_REQUESTED / TERMINATE_REQUESTED |
| `input_data` / `output_data` | `TEXT` (JSON) | 节点输入 / 输出数据 |
| `error_message` | `TEXT` | 错误消息 |
| `intervention_reason` | `TEXT` | 人工介入原因 |
| `execution_order` | `INT` | 执行顺序编号 |
| `plan_version_id` | `BIGINT` | 所属计划版本 ID |
| `branch_key` | `VARCHAR(120)` | 分支标识，如 `"root"` 或 `"root/review-2"` |
| `dynamic_node` | `BOOLEAN` | 是否动态追加节点 |
| `origin_node_name` | `VARCHAR(80)` | 动态节点的原始来源节点名 |
| `last_event_id` | `VARCHAR(64)` | 最近一次事件 ID（SSE 续传用） |
| `state_version` | `BIGINT` | 乐观锁版本号（`@Version`） |
| `started_at` / `completed_at` / `last_attempt_at` / `next_retry_at` | `TIMESTAMP` | 执行时间追踪 |

**节点状态枚举（`TaskNodeStatus`）**：
`PENDING` → `RUNNING` → `SUCCESS` | `FAILED` | `WAITING_RETRY` | `WAITING_INTERVENTION` | `COMPENSATED` | `PAUSED` | `SKIPPED`

**失败分类枚举（`NodeFailureCategory`）**：
`TRANSIENT_INFRASTRUCTURE`（瞬态故障，可重试）| `PERMANENT_BUSINESS`（永久业务错误）| `MANUAL_INTERVENTION_REQUIRED`（需人工介入）| `COMPENSATABLE`（可补偿）| `DOWNSTREAM_CONSUMPTION_GAP`（下游消费缺口）| `UNKNOWN`

**关联**：
- `task_id` → `analysis_task.id`
- `plan_version_id` → `task_plan.id`
- `id` ← `task_node_execution_attempt.node_id`
- `id` ← `agent_execution_log.node_id`

```
analysis_task (1)
  ├── (1:N) task_node
  │       ├── (1:N) task_node_execution_attempt
  │       └── (1:N) agent_execution_log
  ├── (1:N) task_plan
  │       └── (1:N) task_node (via plan_version_id)
  ├── (1:N) evidence_source
  │       └── (1:N) knowledge_document
  │               ├── (1:N) retrieval_index
  │               └── (1:N) retrieval_chunk
  ├── (1:N) competitor_knowledge
  ├── (1:1) report
  │       └── (1:N) report_export_record
  ├── (1:N) task_workflow_event
  ├── (1:N) workflow_dead_letter_record
  ├── (1:N) recovery_checkpoint
  ├── (1:N) memory_snapshot
  ├── (1:N) memory_reuse_record
  └── (1:N) conversation_session
            ├── (1:N) form_draft
            └── (1:N) intent_decision

analysis_schema (独立)
  └── (1:N) analysis_task (via schema_id)

knowledge_domain (独立)
  ├── (1:N) knowledge_document
  └── (1:N) connector_sync_record

organization_quota_snapshot (独立)
connector_runtime_lease (独立)
ai_call_audit_record (独立)
```

> **设计特点**：所有关联通过应用层字段（`task_id`、`node_id`、`plan_version_id` 等）手动管理，不使用 JPA 级联和数据库外键约束。这使得每个表的读写可以完全独立优化，避免了 ORM 级联带来的隐式性能和一致性问题。

#### 关键概念速查

| 概念 | 含义 |
|------|------|
| `AnalysisTask` | 一次竞品分析任务（指定目标产品、竞品名单、分析维度） |
| `TaskNode` | DAG 中的一个执行节点，对应某个 Agent 的一次运行 |
| `AgentType` | 6 种角色：COLLECTOR / EXTRACTOR / ANALYZER / WRITER / CITATION / REVIEWER |
| `AgentContext` | Agent 运行时的"公文包"，携带 taskId、产品信息、共享状态等 |
| `AgentResult` | Agent 执行完的"答卷"，包含 outputData、token 用量、耗时、错误信息 |
| `EvidenceSource` | 一条证据记录（网页抓取结果），带 sourceUrl、可信度评分 |
| `CompetitorKnowledge` | 抽取后的结构化竞品知识（摘要、定位、功能、定价、优劣势） |
| `ExecutionPlanDefinition` | DAG 拓扑定义——哪些节点、谁依赖谁 |
| `WorkflowPlan` | 固化后的执行计划快照 |
| `OrchestrationDecision` | 质量门禁失败后的自动决策 |

**这一阶段你不需要看懂任何 Java 代码**，只需要在脑子里建立"系统里有哪些东西、它们之间什么关系"的概念图。

### 阶段 2️⃣：看"骨架"——理解架构分层

> **目标**：理解 Spring Boot 项目的分层架构和核心抽象
> **预计耗时**：2~3 小时

#### 阅读清单

| 顺序 | 文件 | 为什么读它 | 学什么编程技巧 |
|------|------|------------|---------------|
| 1 | `agent/Agent.java` | Agent 的统一接口——所有 Agent 的"身份证" | **接口隔离原则**：只定义 `getType()` + `execute()`，极简但足够 |
| 2 | `agent/BaseAgent.java` | 模板方法模式——所有 Agent 的公共骨架 | **模板方法模式**：`execute()` 是 final 的，子类只实现 `doExecute()`，自动获得计时 / 日志 / 异常处理 |
| 3 | `agent/AgentContext.java` | 运行时上下文——Agent 之间怎么传数据（sharedState / sharedOutputEnvelopes） | **Builder 模式** + 不可变上下文设计 |
| 4 | `agent/AgentResult.java` | 统一返回值——Agent 怎么输出 | **工厂方法**：`AgentResult.success()` / `.failed()` |
| 5 | `agent/capability/` | 适配器层——`SpringAgentCapabilityRegistry` 把 Spring Bean 和 DAG 执行器解耦 | **适配器模式**：执行器只认 `AgentCapabilityRegistry` 接口 |
| 6 | `workflow/DagExecutor.java` | DAG 执行引擎的心脏——dispatch → await → check → loop | **主循环模式**：线程池 + CompletionService |
| 7 | `controller/` 目录 | REST API 层——对外暴露什么接口 | **薄 Controller 模式**：Controller 只做参数校验和路由，业务逻辑在 Service |

#### 编程方法收获

- **模板方法**：基类定义算法骨架，子类填充具体步骤
- **适配器**：在不修改原有代码的前提下，对接两个不兼容的接口
- **Builder**：构建复杂对象，链式调用，代码可读
- **工厂方法**：用静态方法创建对象，语义更清晰

这些都是企业级 Java 最常用的设计模式。

## 💡 这个项目最值得学的编程思想

### 1. "不信任 AI"的工程设计

每个 Agent 都输出 `issueFlags`、`evidenceFragments`、`gapComment`、`sourceUrls`。不管 LLM 输出什么，系统都能追溯到原始证据。例如：

- 抽取 Agent 对每个字段标注证据覆盖状态：`TRACEABLE` / `MISSING_EVIDENCE` / `LLM_REFUSED`
- 评审 Agent 检查报告的"结论""建议""风险点"段落是否都有 `[evidence: E001]` 标记
- 引用核查 Agent 用纯规则评估每条证据来源的可信度

### 2. 模板方法 + 适配器的组合

`BaseAgent.execute()` 是 final 的模板方法，所有 Agent 自动获得：

- TraceId 注入（可观测性）
- 前置上下文组装（RAG 上下文注入）
- 执行计时
- 异常 → `AgentResult.failed()` 翻译
- 执行日志持久化
- 上下文清理

而 `SpringAgentCapabilityRegistry`（适配器）让 DAG 执行器只依赖 `AgentCapabilityRegistry` 接口，未来换执行器不需要改任何 Agent 代码。

### 3. Markdown 也是机器可读的协议

Writer 输出的报告里带 `[evidence: E001]` 标记，CitationAgent 用正则解析 Markdown 来做引用核查。**人类可读和机器可解析并不矛盾**——设计好约定即可。

### 4. 规则引擎兜底

编排决策层（`OrchestrationDecisionService`）明确注释：

> P1 不做 LLM 调用，纯规则保证 demo 版本稳定可复现

在关键路径上，**确定性比智能更重要**。把 LLM 用在创意性工作（分析、撰写），把流程控制交给确定性规则。

### 5. 证据溯源全链路

`EvidenceFragment` → `SectionEvidenceBundle` → `DownstreamEvidenceView` 三个类型贯穿所有 Agent：

```
Collector 产出            Extractor 消费 + 补充        Analyzer 再次消费 + 补充      Writer 最终使用
原始网页 + sourceUrl  →  字段级证据覆盖状态标注  →  分析结论绑定证据片段    →  报告内嵌 [evidence: EID]
```

这条"证据链"是整个系统最重要的数据流，保证了**所有分析结论有据可查**。
