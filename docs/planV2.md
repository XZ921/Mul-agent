# AI 驱动的竞品分析 Agent 协作系统 V2 项目计划

## 1. V2 项目核心目标

V1 已经完成了“创建任务 -> 执行固定工作流 -> 生成可溯源 Markdown 报告 -> 查看日志与证据”的主链路验证，并已完成测试与基础联调。

V2 的目标不是重新做一套系统，而是在当前工程基础上，把它从“可演示 MVP”升级为“可持续迭代、可恢复、可校验、可补源、可闭环修订”的研究工作流。

V2 的核心目标包括：

1. 让任务执行从“固定流程”升级为“更可配置的动态工作流”。
2. 让信息采集从“用户提供 URL 为主”升级为“系统能主动补源与筛选”。
3. 让证据追溯从“字段级初步可看”升级为“章节级、结论级可校验”。
4. 让质检从“只给分”升级为“可驱动修订与复核的闭环”。
5. 让任务恢复从“整任务重试”升级为“节点级恢复、节点级重跑、人工干预编排”。
6. 让报告交付从“Markdown 为主”升级为“更正式的导出与展示能力”。
7. 让系统可观测性从“日志可看”升级为“过程、质量、证据、修订链路完整可回放”。

## 2. V2 开发前提与当前基线

V2 计划基于当前已完成的实际能力展开，而不是从空白重新规划。

当前已经具备的基线能力包括：

1. 前后端分离工程结构已经稳定，后端与前端都可独立运行。
2. 基础任务管理、任务详情、报告预览、Agent 日志查看流程已经打通。
3. 采集、抽取、分析、撰写、质检五类 Agent 已有工程实现。
4. DAG 节点、任务状态、日志、证据、报告等核心数据模型已落库。
5. 报告支持 Markdown 预览与导出。
6. 节点追踪、输入输出查看、上游依赖查看、证据追踪等面板已具备第一版能力。
7. 已具备 `quality_check -> rewrite_report -> quality_check_final` 的闭环基础。
8. 已具备 `resumeTask` 与指定节点 `rerun` 的第一版能力。
9. 已具备启发式信息源发现能力，可基于官网 URL 或竞品名扩展 docs / pricing / news / review 入口。

因此，V2 的重点不是补 V1 的基础 CRUD，而是把已有的 V2 第一版能力做深、做全、做稳。

## 3. V2 用户使用流程

V2 用户流程以“任务创建 -> 自动补源与动态规划 -> 自动执行 -> 质检修订闭环 -> 节点级恢复/重跑 -> 正式导出”为主。

### 3.1 创建竞品分析任务

用户创建任务时，需要填写：

1. 分析主题。
2. 本方产品或关注对象。
3. 竞品列表。
4. 分析维度。
5. 采集范围，例如官网、文档、定价、新闻、测评。
6. Schema 或分析模板。
7. 报告语言与报告模板。

### 3.2 系统生成执行计划

系统根据任务输入生成本次分析计划，包括：

1. 需要补充的候选信息源。
2. 每个竞品对应的采集节点。
3. 抽取、分析、撰写、质检、重写、终审等节点。
4. 节点依赖关系与执行顺序。
5. 每个节点的配置摘要。

用户可以在任务启动前预览本次计划，并查看补源说明。

### 3.3 多 Agent 自动执行

任务启动后，系统按依赖顺序执行：

1. 自动发现和补充候选信息源。
2. 采集公开资料并保存元数据。
3. 抽取竞品知识 Schema，并补齐字段级证据覆盖情况。
4. 生成结构化竞品横向分析结果。
5. 生成正式报告。
6. 执行初审。
7. 如初审失败，则生成修订计划并触发重写。
8. 重写后执行终审。

### 3.4 查看执行过程与人工干预

用户可以在任务详情页查看：

1. 任务整体状态与当前阶段。
2. 节点执行状态、依赖关系、输入输出、节点配置。
3. 已采集信息源、证据数量、来源类型与补源说明。
4. 初审结果、修订计划、重写结果、终审结果。
5. 失败节点与错误摘要。

V2 需要支持的人工操作包括：

1. 恢复任务。
2. 指定节点重跑。
3. 对下游节点做选择性失效并重新执行。
4. 在必要时修改节点配置后继续执行。

### 3.5 查看和导出报告

任务完成后，用户可以查看：

1. 报告正文。
2. 初审与终审结果。
3. 修订计划与是否已应用。
4. 证据列表。
5. 竞品知识结构化结果及字段级证据覆盖。

V2 优先支持：

1. Markdown 导出。
2. HTML 导出。
3. PDF 导出。

Word 与 PPT 导出放入后续版本。

## 4. V2 核心功能范围

### 4.1 执行器与任务编排升级

V2 需要在当前执行器基础上补齐以下能力：

1. 采集节点按“竞品 + sourcePlan”动态展开。
2. 工作流节点具备更明确的 `dependsOn`、`required`、`retryable`、`maxRetries`、`executionOrder` 语义。
3. 支持节点级恢复，而不是只能整任务重跑。
4. 支持从指定节点开始重跑，并自动识别受影响的下游节点。
5. 为后续按任务类型、模板、Schema 自动规划完整 DAG 做好扩展准备。

V2 不要求一步做到完全通用的 DAG 平台，但要求当前工作流不再被固定五节点设计卡死。

### 4.2 信息源发现与采集升级

V2 需要让系统具备更强的补源能力：

1. 在用户输入官网 URL 的基础上自动扩展文档页、定价页、博客/新闻页、测评页。
2. 增加搜索式补源能力，而不只依赖启发式路径拼接。
3. 增加来源去重能力，避免同一来源被重复采集。
4. 增加来源排序能力，包括相关性、时效性、来源质量。
5. 增加采集元数据，包括来源类型、发布时间、采集时间、原文片段、来源说明。

V2 第一优先级是把补源做“可用”，而不是一次做到复杂的搜索平台。

### 4.3 Schema 与证据约束升级

V2 需要把当前字段级追溯继续强化：

1. 继续使用结构化 Schema 作为抽取与分析的核心载体。
2. 为关键字段保留 `sourceUrls`、`evidenceIds`、`evidenceCoverage` 等结构。
3. 增强字段级证据状态判断，区分“完整支持”“部分支持”“无证据支持”。
4. 增加章节级证据约束。
5. 增加结论级证据约束。
6. 让质检阶段能识别“结论存在但证据不足”的问题。

V2 不要求所有字段都做到完全自动校验，但要求关键章节和核心结论不能继续停留在弱约束状态。

### 4.4 分析、撰写与质检闭环

V2 需要让闭环能力更稳定、更可控：

1. 分析 Agent 输出结构化分析结果，不退化成纯大段文本。
2. Writer 支持首稿与修订稿两种工作模式。
3. Reviewer 输出评分、问题列表、问题级别与修订计划。
4. 当初审失败时，系统能触发 Writer 根据修订计划重写。
5. 重写后系统能执行终审，并将结果纳入最终任务状态判断。
6. 对缺证据章节、缺证据结论、空泛结论增加更严格的校验逻辑。

V2 的目标是把“可修订闭环”做稳定，而不是引入过多全新的 Agent 角色。

### 4.5 报告与导出能力

V2 报告页需要从“预览结果”升级为“查看生产过程 + 查看最终交付”：

1. 报告页展示初审结果、修订计划、终审结果。
2. 报告页展示是否发生重写，以及最新质量反馈。
3. 报告页展示证据列表与竞品溯源摘要。
4. 保留 Markdown 与 HTML 导出。
5. 补齐 PDF 导出。
6. 导出内容保留证据、质量状态、修订结果和溯源摘要。

### 4.6 可观测性与质量追踪

V2 需要继续增强系统可观测性：

1. 任务级状态、节点级状态完整入库。
2. Agent 输入、输出、模型名、耗时、token、prompt、错误原因持续记录。
3. 节点追踪面板展示配置、输入输出、依赖链与证据情况。
4. 记录初审、修订、终审三个阶段的质量数据。
5. 为后续质量看板与成本统计预留数据基础。

### 4.7 前端体验升级

V2 前端需要重点补强以下能力：

1. 新建任务页支持更清晰的采集范围、Schema、报告模板配置。
2. 任务详情页支持动态节点、节点配置查看、恢复、重跑与状态说明。
3. 报告页支持闭环视图、证据展开、修订计划查看、导出入口。
4. 对缺证据字段、缺证据章节、失败节点给出更清晰提示。

## 5. V2 暂不纳入范围的内容

为了避免 V2 范围继续失控，以下能力不纳入本阶段：

1. 用户登录、权限系统、团队空间。
2. 私有知识库深度集成。
3. 分布式任务编排。
4. PPT 自动生成。
5. 全量 BI 仪表盘。
6. 复杂多 Agent 辩论式分析。

这些内容可以作为 V3 或更后续版本规划。

## 6. V2 分阶段开发计划

V2 建议按 5 个阶段推进，每个阶段都必须能单独验收，而不是把所有能力堆到最后统一联调。

### 第 1 阶段：V2 基线收口与任务编排稳固，0.5-1 天

目标：

1. 明确当前 V2 基线能力与边界。
2. 收口动态采集节点、节点追踪、恢复与重跑语义。
3. 固化当前闭环工作流与任务状态定义。

主要工作：

1. 整理 V2 计划文档与当前实现边界。
2. 校验 `resumeTask`、`rerunFromNode`、节点依赖解析与下游失效逻辑。
3. 补充节点级重跑、恢复相关测试。
4. 校验前端任务详情页对动态节点与节点追踪信息的展示。

### 第 2 阶段：搜索式补源与来源筛选，1-1.5 天

目标：

1. 从启发式补源升级到“启发式 + 搜索式”的补源能力。
2. 让采集来源具备可筛选、可排序、可去重能力。
3. 提升抽取与分析的原始信息质量。

主要工作：

1. 增加搜索式来源发现接口或适配层。
2. 引入来源去重逻辑。
3. 引入来源排序规则，包括相关性、时效性、来源质量。
4. 扩展来源元数据结构。
5. 在任务节点配置和详情页中展示补源说明。

### 第 3 阶段：章节级证据强校验与修订闭环增强，1 天

目
1. 让质检不再只检查格式完整性。
2. 对关键章节和关键结论建立证据硬约束。
3. 让修订计划能更精准地驱动重写。

主要工作：

1. 为关键章节建立证据覆盖规则。
2. 为关键结论建立证据可回指规则。
3. 增强 Reviewer 对证据不足问题的识别与分级。
4. 增强 Writer 对修订计划的消费逻辑。
5. 在报告页和任务详情页增加缺证据提示。

### 第 4 阶段：任务规划增强与人工干预完善，0.5-1 天

目标：

1. 让工作流不仅能动态展开采集节点，还能更好表达人工干预能力。
2. 为后续完整 Schema 驱动 DAG 规划打基础。

说明：

1. 本阶段节点级恢复、重跑、修改配置后继续、停止任务、检查点复用与能力边界说明，统一收口在 `phase4-intervention-rules.md`。
2. 如第四阶段人工干预化语义发生变，必须同步更新 `phase4-intervention-rules.md`，避免代码与文档脱节。

主要工作：

1. 完善从指定节点重跑的下游失效策略。
2. 支持必要的人工修改配置后继续执行。
3. 补充节点暂停、跳过、强制终止等能力评估与设计。
4. 优化任务计划预览与节点配置展示。

### 第 5 阶段：正式导出与演示打磨，0.5-1 天

目标：

1. 补齐 PDF 导出。
2. 打磨报告页与导出页的一致性。
3. 固化 V2 演示用例与测试回归流程。

主要工作：

1. 实现 PDF 导出。
2. 统一 Markdown / HTML / PDF 导出内容结构。
3. 补充闭环任务样例。
4. 回归测试“创建任务 -> 自动补源 -> 执行 -> 修订 -> 终审 -> 导出”主流程。

## 7. V2 每个阶段验收标准

### 第 1 阶段验收标准

1. `planV2.md` 与当前实现边界一致。
2. 动态采集节点、修订闭环、恢复与重跑语义文档化完成。
3. `resumeTask` 与指定节点 `rerun` 具备测试覆盖。
4. 任务详情页可稳定展示动态节点、依赖关系、配置、输入输出与证据追踪。

### 第 2 阶段验收标准

1. 系统可基于启发式与搜索式能力补充候选来源。
2. 候选来源具备去重能力。
3. 候选来源具备基础排序能力。
4. 采集结果中能看到来源类型、来源说明、发布时间或等价元数据。
5. 补源结果可在节点配置或任务详情页中查看。

### 第 3 阶段验收标准

1. Reviewer 能识别章节级缺证据问题。
2. Reviewer 能识别关键结论缺证据问题。
3. 修订计划中能明确指出需要补强的章节或结论。
4. Writer 能根据修订计划输出更有针对性的修订稿。
5. 报告页能清晰展示初审、修订、终审和最新质量结果。

### 第 4 阶段验收标准

1. 用户可以从指定节点开始重跑任务。
2. 系统能自动识别并重置受影响的下游节点。
3. 成功节点产物能够在恢复或重跑时复用。
4. 前端提供明确的恢复与重跑入口。
5. 节点级人工干预规则在代码和文档中保持一致。

### 第 5 阶段验收标准

1. Markdown、HTML、PDF 三种导出方式可用。
2. 导出结果包含报告正文、质量状态、修订结果和证据摘要。
3. 至少准备一个稳定可演示的 V2 闭环任务案例。
4. 已完成从后端到前端的完整回归测试。
5. V2 主流程可以重复跑通并用于对外演示。

## 8. V2 完成标准

V2 完成时，应满足以下整体标准：

1. 用户可以创建任务并看到动态展开的执行计划。
2. 系统可以自动补源、采集、抽取、分析、撰写、质检、重写和终审。
3. 用户可以查看节点级追踪、证据追踪、修订计划与质量结果。
4. 用户可以恢复任务、从指定节点重跑，并复用已有成功节点结果。
5. 报告关键章节和关键结论具备更明确的证据约束。
6. 报告支持 Markdown、HTML、PDF 导出。
7. V2 主流程具备可重复运行、可测试、可演示的稳定性。
