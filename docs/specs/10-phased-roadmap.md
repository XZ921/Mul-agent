# 10 Phased Roadmap

## 文档目的

本文档用于描述系统如何从当前工程基线逐步演进到长期蓝图，但它不替代蓝图本身。路线图必须服务于 `specs` 目录中定义的长期稳定原则、能力域边界和实施约束，而不是服务于短期实现便利。

## 当前工程基线

结合当前仓库现状，系统已经具备的基础包括：

1. 前后端分离的工程结构。
2. 基础任务管理、任务详情、报告预览和日志查看能力。
3. 采集、抽取、分析、撰写、质检五类 Agent 的第一代实现。
4. 以 `DagExecutor` 为核心的增强型工作流执行能力。
5. 节点、日志、证据、报告等基础数据落库能力。
6. 多源补源、来源治理、基础质量闭环和报告追溯的第一代能力。
7. `Redis` 进度快照、最小并发保护、`SSE` 实时观察主路径和基础恢复语义。
8. 多供应商配置驱动的 `infra-ai` 雏形。

## 终态蓝图与现状差距

当前工程与长期蓝图之间的主要差距包括：

1. 任务建模仍偏向单一竞品分析场景。
2. 前端仍缺少正式的产品化治理约束，工作台容易退化为技术观察台。
3. 任务图仍偏向增强型同步 `DAG`，尚未进入正式事件驱动骨架。
4. `Task RAG`、记忆、上下文引擎尚未成为正式底座。
5. 对话统一入口、意图识别和动作路由尚未系统化。
6. `infra-ai` 仍处于配置驱动雏形阶段，尚未形成正式治理层。
7. 组织级知识、正式交付、审计回放和平台治理能力尚未收口。

## 阶段划分原则

1. 每个阶段都应独立形成可验证能力，而不是只做底层铺垫。
2. 先补实施治理与产品骨架，再扩复杂能力；不能在失控的前端和体验底座上继续叠加高级能力。
3. 对系统结构产生质变的骨架能力，应优先于锦上添花的附加能力。
4. 阶段内允许做取舍，但不得破坏证据可追溯、任务可恢复、工作台可理解和统一对话入口等长期方向。

## 阶段目标

### 第一阶段：从可用工作流升级为高质量多源研究引擎与轻量运行底座

目标：

1. 让网址和本方产品从硬依赖升级为可选输入。
2. 让系统具备更稳定的多源补源、来源治理和质量闭环能力。
3. 为后续产品化工作台建立 `Redis + SSE + 最小恢复语义` 的轻量底座。
4. 为后续对话入口、任务骨架和 AI 能力治理保留扩展边界。

核心建设项：

1. 任务输入模型升级
2. 搜索、补源、来源元数据与来源治理增强
3. 章节级和结论级质量闭环的第一版收口
4. `Redis` 快照、中间态保护、基础锁与幂等保护
5. `SSE` 实时观察主通道与断线补偿基础
6. `infra-ai` 第一版抽象

前置依赖：

1. 保持现有执行器与数据模型可演进。
2. 收口动态节点、重跑和恢复语义。

非目标：

1. 不要求完整产品化工作台重建。
2. 不要求正式 `RocketMQ` 异步编排骨架。
3. 不要求统一对话入口深度接管任务动作。

### 第二阶段：从高质量研究引擎升级为受治理的产品化工作台

目标：

1. 建立前端产品化治理基线，避免工作台继续退化为调试面板集合。
2. 重建任务创建工作台、执行控制台和报告交付中心的信息架构。
3. 建立用户语言、统一状态提示、问题处理路径和高级诊断边界。
4. 让后续所有复杂能力都长在稳定、可审计、可理解的产品骨架上。

核心建设项：

1. 前端产品化与设计治理规范落地
2. 三大工作台信息架构重建
3. 用户语言与技术语义翻译层
4. 调试信息、高级诊断和原始结构体的受控披露机制
5. 问题处理导向的质量诊断、证据追溯和人工干预体验
6. 面向用户的运行观察面与统一状态反馈体系

前置依赖：

1. 第一阶段完成 `SSE` 主路径、`Redis` 快照和基础恢复语义。
2. 第一阶段完成来源治理和质量闭环的结构化语义收口。
3. 系统已具备可继续产品化的最小任务主链路。

非目标：

1. 不要求在本阶段引入正式 `RocketMQ` 异步骨架。
2. 不要求完整统一对话入口深度接管。
3. 不要求组织级知识平台、连接器平台和多租户治理。

### 第三阶段：从增强型工作流升级为事件驱动任务图骨架

目标：

1. 让系统从增强型同步 `DAG` 升级为正式事件驱动任务图骨架。
2. 让证据缺口、质量问题和补偿逻辑能够驱动动态补图、回流和局部恢复。
3. 在长耗时、多节点、多任务场景下建立正式异步调度、重试和死信处理能力。
4. 以最小可用方式引入 `Task RAG` 和解释型统一对话入口，为后续智能能力铺路。

核心建设项分层如下：

#### 必须交付，不可降级

1. `RocketMQ` 异步编排正式落地
2. 事件驱动的节点状态流转、重试、`DLQ` 与补偿语义
3. 动态补图、回流和局部恢复增强
4. 从“增强型同步 `DAG`”升级为“正式任务图骨架”

#### 应该交付，允许 MVP 级

1. `Task RAG` 最小可用版
2. 任务级向量索引、基础召回和简单重排
3. 统一对话入口雏形，但以“解释型入口”为主
4. 任务级上下文装配的第一版正式链路

#### 允许下沉到下一阶段

1. 完整意图识别与任务动作路由
2. 长短期记忆深度融合
3. 更成熟的跨任务知识复用
4. 更强的对话驱动任务控制

前置依赖：

1. 第二阶段已形成受治理的产品化工作台骨架。
2. 第一阶段与第二阶段已收口任务状态、人工干预、质量诊断和用户语言语义。
3. 平台具备承接正式异步调度和动态任务图的基础存储、事件和观察条件。

非目标：

1. 不要求在本阶段完成组织级知识平台最终形态。
2. 不要求完整的模型网关治理平台。
3. 不要求统一对话入口已经深度接管复杂动作控制。

### 第四阶段：从任务图骨架升级为企业级治理与正式交付系统

目标：

1. 建立组织级知识、正式交付和平台治理能力。
2. 完成 `infra-ai` 从能力抽象雏形到正式治理层的升级。
3. 吸收上一阶段下沉的对话、记忆和知识复用能力。
4. 收口审计回放、恢复平台、正式导出和运营治理能力。

核心建设项：

1. `Domain RAG / Organization RAG`
2. 长短期记忆深度融合与组织级知识演进
3. 完整意图识别与任务动作路由
4. `Model Gateway`、多供应商治理、限流、熔断和成本治理
5. 正式的任务回放、恢复和审计平台
6. 正式导出、交付规范与组织级扩展入口

前置依赖：

1. 第三阶段已经形成正式事件驱动任务图骨架。
2. 第二阶段已经形成稳定工作台与产品化治理约束。
3. 平台已经具备承接组织级治理要求的业务广度和技术深度。

非目标：

1. 不要求在一开始就开放无边界第三方插件生态。
2. 不要求优先建设大而全 BI 运营平台。

## 每阶段非目标

除上文阶段内非目标外，整个路线图长期不建议在早期纳入：

1. 复杂权限与团队空间大一统改造
2. 通用开放式聊天产品化
3. 无边界第三方插件生态
4. 与阶段主骨架无关的大规模局部美化工程

## 阶段验收标准

### 第一阶段验收标准

1. 任务输入不再依赖网址和本方产品必填。
2. 补源结果、来源元数据和质量问题具备结构化语义。
3. `Redis` 快照、基础幂等和最小恢复语义进入正式主链路。
4. 前端观察主路径完成从轮询向 `SSE` 的切换。

### 第二阶段验收标准

1. 三大工作台形成统一的信息架构与用户语言体系。
2. 默认界面不再以原始技术字段、原始 JSON 和调试信息作为主内容。
3. 问题、动作、证据追溯和质量诊断形成以处理问题为中心的主路径。
4. 前端产品化治理规范进入正式验收口径。

### 第三阶段验收标准

1. 长耗时任务正式摆脱强同步串行主路径。
2. 任务图具备动态补图、回流和局部恢复能力。
3. `RocketMQ`、重试、`DLQ` 和补偿语义进入正式链路。
4. `Task RAG` 和解释型统一对话入口至少达到 MVP 可用水平。

### 第四阶段验收标准

1. `infra-ai` 完成从抽象雏形到正式治理层的升级。
2. 组织级知识、审计回放、恢复平台和正式导出形成闭环。
3. 上一阶段下沉的对话、记忆和知识复用能力完成正式收口。
4. 系统整体具备“企业级研究任务操作系统”的核心轮廓。

## 风险与回退策略

1. 若某阶段能力过大，应先做阶段内优先级分层，而不是压缩长期原则。
2. 若某项新需求明显改变蓝图，应先更新相关能力域文档，再调整本路线图。
3. 若短期实现方案与 `07`、`12` 定义的产品化治理约束冲突，应优先保留扩展边界和治理规则，而不是为了局部交付继续放大历史问题。

## 当前结论

1. 路线图不是最终计划本体，而是蓝图和实施约束的落地映射。
2. 从当前基线继续前进时，下一步优先级不是再堆新能力，而是先建立产品化治理与工作台骨架。
3. 真正让系统发生结构质变的关键阶段，是“事件驱动任务图骨架成型阶段”，该阶段的骨架能力优先级高于附加能力。

## 扩展预留

1. 后续可在第四阶段之后继续新增“组织治理深化”“协作审批”“连接器生态”等阶段。
2. 若未来出现新的一级能力域，可新增对应 roadmap 子阶段说明。

## 待确认问题

1. 是否需要把部分产品化治理规则进一步自动化为工程检查。
2. 是否需要在第三阶段前置部分任务模板和行业方法论资产，以增强 `Task RAG` 的可用性。

## 本阶段不约束的内容

1. 不要求路线图细化为每周执行计划。
2. 不要求每个阶段现在就绑定明确时间承诺。
