# 07 Frontend Workbench And Delivery Center

## 文档目的

本文档用于定义系统最终前端工作台的信息架构、用户默认视图和展示纪律，包括任务创建工作台、执行控制台、报告交付中心以及页面内对话助手之间的协同关系。

本文档回答的是“用户应该在界面里先看到什么、如何理解问题、如何完成动作”；实现治理和验收门槛由 `12-frontend-productization-and-design-governance.md` 定义。

## 本能力域在系统中的职责

本能力域负责把后端的任务、证据、执行与交付能力转化为用户可理解、可操作、可干预的统一界面系统。

## 业务目标与用户价值

1. 让用户容易创建复杂研究任务。
2. 让用户理解执行过程与风险点。
3. 让报告、诊断、证据与动作建议在一个交付界面里闭环。

## 当前结论

最终交互层应由三大工作台和一个统一对话入口组成：

1. 任务创建工作台
2. 执行控制台
3. 报告交付中心
4. 页面内对话助手

并且应遵守以下总原则：

1. 一级界面围绕“任务目标、当前问题、建议动作、证据追溯”组织。
2. 技术实现细节允许存在，但必须作为高级诊断能力后置。
3. 工作台首先服务业务理解与决策，而不是服务开发调试。

## 目标能力

1. 任务表单与对话双入口
2. 动态任务图与节点诊断视图
3. 证据覆盖与缺口可视化
4. 质量 checkpoint 与修订计划展示
5. 节点级动作与人工干预入口
6. 用户语言与技术语义之间的稳定翻译层
7. 基于问题处理的统一首屏结构

## 关键场景

1. 用户通过表单或对话快速建立任务。
2. 用户在执行控制台查看并行分支、节点状态和工具轨迹。
3. 用户在报告页查看正式报告、质量结果和证据覆盖。
4. 用户在页面内通过自然语言执行解释或动作。

## 全局信息架构原则

### 1. 页面首屏必须优先回答的问题

所有核心工作台在首屏默认状态下，都应尽量回答以下问题中的大部分：

1. 我当前在处理什么任务或结果。
2. 系统现在进行到哪一步。
3. 当前最值得关注的问题是什么。
4. 我下一步可以执行什么动作。
5. 如果我要追溯依据，应该去哪里看。

### 2. 默认展示层级

对大多数工作台，推荐采用以下信息层级：

1. 任务目标或交付目标
2. 当前阶段与状态摘要
3. 问题列表、风险提示与下一步动作
4. 关键结果、关键证据、关键诊断
5. 高级诊断、原始配置、原始输入输出、事件级技术细节

### 3. 用户语言优先

主界面中的主标题、一级区域名和主操作应优先使用业务语言，而不是底层技术术语。例如：

1. `DAG` 更适合作为“任务执行路径”或“任务图”被解释，而不是直接占据主标题。
2. `SSE` 更适合作为“实时进度通道”被内部承载，而不是直接占据主提示语言。
3. `fallback` 更适合作为“已切换为兜底模式”被解释，而不是直接暴露给用户。

### 4. 渐进披露

原始配置、原始输入输出、事件回放、技术 trace 和其他调试信息属于高级诊断能力，必须后置到显式展开视图，不得构成首屏默认体验。

## 核心对象 / 核心模块

### 任务创建工作台

负责录入任务目标、研究对象、分析维度、资料策略、输出要求，并在任务启动前展示结构化计划预览。

#### 用户目标

1. 明确“我要研究什么、想产出什么”。
2. 理解系统准备如何执行这次研究。
3. 在启动前修正范围、来源策略和输出预期。

#### 默认首屏应优先展示

1. 任务目标与研究对象
2. 分析维度与输出要求
3. 资料策略摘要
4. 执行计划预览
5. 补源说明与预期来源范围摘要

#### 默认不应直接堆叠展示

1. 原始节点配置 JSON
2. 原始搜索 Query 列表
3. 原始候选来源 URL 明细
4. 技术枚举值和内部布尔开关集合

#### 任务创建页的核心原则

1. 用户先理解“这次任务会怎么做”，再决定是否启动。
2. 计划预览必须是业务可读的，而不是节点字段平铺。
3. 资料策略应以来源类型、范围、质量要求来表达，而不是以技术开关主导表达。

### 执行控制台

负责展示任务执行路径、当前阶段、关键风险、建议动作和人工干预入口，并在需要时提供高级追踪能力。

#### 用户目标

1. 快速知道任务整体现在进行到哪里。
2. 发现阻塞点、失败点和证据缺口。
3. 在恰当位置执行暂停、恢复、重跑、改配置等动作。

#### 默认首屏应优先展示

1. 当前阶段与整体健康状态
2. 分支进度或任务图摘要
3. 当前主要问题与阻塞点
4. 可执行动作
5. 活跃中的来源活动、节点活动或质量告警摘要

#### 次级层可展示

1. 节点摘要列表
2. 来源治理进度
3. 搜索活动摘要
4. 质量诊断摘要
5. 证据缺口摘要

#### 高级诊断层才展示

1. 节点原始输入输出
2. 原始事件轨迹
3. 技术状态机明细
4. 工具级 trace
5. 原始日志流细节

#### 执行控制台的核心原则

1. 页面中心不是“列出所有节点字段”，而是“解释为什么当前在这里、哪里有问题、用户接下来能做什么”。
2. 人工干预入口必须与对应问题一起出现，不能要求用户先理解完整内部编排结构。
3. 实时观察能力必须服务任务理解，而不是变成一块只供开发者阅读的链路大屏。

### 报告交付中心

负责展示正式报告、质量结论、问题清单、证据追溯和导出动作，让用户在一个界面内完成“阅读结果 + 处理问题 + 追溯依据”。

#### 用户目标

1. 先看到当前可交付结果是什么。
2. 再理解质量是否通过、问题在哪里、建议如何处理。
3. 最后在需要时继续追证、修订或导出。

#### 默认首屏应优先展示

1. 报告正文或最新可交付稿
2. 质量结论与问题摘要
3. 修订建议与推荐动作
4. 关键章节证据覆盖情况
5. 导出入口

#### 次级层可展示

1. 章节级证据覆盖
2. 关键结论的来源追溯
3. 修订历史摘要
4. 审计摘要

#### 高级诊断层才展示

1. 原始评审结构体
2. 原始知识覆盖 JSON
3. 原始评分因子与详细计算过程
4. 深层技术审计明细

#### 报告交付中心的核心原则

1. 报告页不是“文档预览器”，而是“结果交付与问题处理中心”。
2. 质量结果必须被转化为可理解、可执行的用户动作。
3. 证据追溯必须对业务用户可读，而不是要求用户阅读原始结构体。

### 页面内对话助手

负责在页面上下文内提供解释、填表、动作和任务协助能力。

#### 页面内对话助手的默认职责

1. 解释当前页面、当前阶段和当前问题。
2. 帮助用户理解为什么系统会这样执行。
3. 在能力边界允许时辅助填表或触发动作。

#### 页面内对话助手的边界

1. 它应服务当前页面与当前任务，而不是变成脱离上下文的通用聊天窗口。
2. 它的动作能力应遵守 `06-conversation-intent-and-task-orchestration.md` 的模式边界。
3. 它的呈现方式也应遵守本文件和 `12` 中关于用户语言与渐进披露的要求。

## 核心流程 / 状态流 / 编排关系

1. 用户在创建工作台定义任务并确认计划。
2. 执行控制台跟随任务执行持续刷新状态。
3. 报告交付中心在关键结果可用后聚合展示。
4. 页面内对话助手贯穿上述三个工作台并共享上下文。

## 关键约束与设计原则

1. `网址与本方产品应改为选填`
   前端表单不应把短期实现习惯固化为长期业务约束。
2. `计划与进度必须可视化`
   用户需要理解系统准备怎么做和正在怎么做。
3. `交付页不是纯结果查看器`
   它必须同时承担诊断与下一步动作指引。
4. `技术细节必须后置`
   任何原始结构和链路明细都不应成为默认体验主内容。
5. `工作台必须以问题处理为中心`
   页面默认结构应围绕目标、状态、问题、动作和追溯组织，而不是围绕技术字段组织。
6. `对话不能跳出当前系统`
   页面内对话能力应服务当前任务与页面，而不是外跳通用聊天。

## 与其他能力域的边界

1. 本文档定义信息架构、默认视图和展示纪律，不展开具体组件实现细节。
2. `12-frontend-productization-and-design-governance.md` 定义实现治理、评审门槛和强制约束。
3. 页面内对话的意图和模式由 `06-conversation-intent-and-task-orchestration.md` 定义。
4. 执行图语义由 `04-agent-runtime-and-workflow-architecture.md` 定义。
5. 质量对象、修订计划和审计语义由 `09-quality-evidence-and-audit.md` 定义。

## 扩展预留

1. 后续可增加上传资料中心、知识资产面板和团队协作区。
2. 后续可增加更强的报告诊断视图和对比视图。
3. 后续可增加组织模板市场与任务模版库。

## 待确认问题

1. 执行控制台是否需要长期拆分为“任务驾驶舱”和“节点深度诊断台”两层。
2. 报告交付中心是否需要为不同任务类型提供差异化布局。

## 本阶段不约束的内容

1. 不规定当前必须采用哪一种具体图形可视化组件。
2. 不要求所有页面在当前版本一次性实现终态信息密度。
3. 不要求所有高级诊断能力都在默认主路径暴露。
