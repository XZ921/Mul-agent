# 02 User Journeys And Task Modes

## 文档目的

本文档用于定义用户从建任务到查看交付物的完整旅程，以及统一对话入口下的多模式路由规则。

## 本能力域在系统中的职责

本能力域负责回答两个问题：

1. 用户在系统里如何完成一个研究任务。
2. 用户的自然语言输入在不同场景下应该被系统识别成什么模式。

## 业务目标与用户价值

1. 降低任务创建门槛。
2. 让用户能够理解系统当前正在做什么。
3. 让用户能够在必要时接管、调整或恢复任务。
4. 让报告交付和任务诊断形成统一体验。

## 当前结论

最终系统应长期支持以下五条主旅程：

1. 表单创建任务
2. 对话创建任务
3. 执行中观察与人工干预
4. 报告交付与诊断
5. 对话式解释与动作执行

## 目标能力

1. 单一对话入口
2. 多种任务模式内部路由
3. 页面感知式对话上下文
4. 计划预览与进度可视化
5. 人工干预动作可追溯
6. 面向问题处理的用户界面主路径

## 关键场景

### 场景一：表单创建任务

用户通过结构化表单录入任务目标、对象、维度、资料策略和输出要求，系统在启动前生成执行计划预览。

在这个旅程里，用户默认首先应看到：

1. 任务目标与研究范围
2. 系统准备如何执行
3. 资料策略和补源说明摘要
4. 启动前可调整的关键输入

用户不应被迫先阅读：

1. 原始节点配置
2. 原始搜索 Query 明细
3. 内部技术枚举与调试数据

### 场景二：对话创建任务

用户用自然语言描述需求，系统先识别任务意图，再自动填充结构化字段，并通过追问补齐必要槽位。

在这个旅程里，系统应优先帮助用户：

1. 明确任务目标和对象
2. 补齐必要槽位
3. 理解即将执行的研究路径

### 场景三：执行中观察与人工干预

用户查看动态任务图、当前阶段、节点状态、证据缺口和工具轨迹，并在必要时执行暂停、恢复、重跑或改配置。

在这个旅程里，用户默认首先应看到：

1. 当前任务进行到哪一步
2. 当前最值得关注的问题是什么
3. 当前可以执行什么动作
4. 需要时到哪里查看依据和更深层诊断

用户不应被迫先阅读：

1. 节点原始输入输出
2. 原始事件流和技术 trace
3. 原始状态机细节
4. 只有开发者才理解的技术术语

### 场景四：报告交付与诊断

用户查看正式报告、证据覆盖、质量 checkpoint、修订计划和导出物，不必在多个页面之间切换。

在这个旅程里，用户默认首先应看到：

1. 当前最新可交付结果
2. 质量是否通过
3. 当前最需要处理的问题和建议动作
4. 关键结论的证据去哪里看

用户不应被迫先阅读：

1. 原始评审结构体
2. 原始知识覆盖 JSON
3. 技术级评分细节

### 场景五：对话式解释与动作执行

用户在任何页面都可以询问“这个页面是什么”“现在做到哪一步了”“从这个节点重跑”等问题，由系统识别为解释或动作请求。

在这个旅程里，对话首先服务于：

1. 解释当前任务、当前页面和当前问题
2. 降低用户理解成本
3. 在边界允许时辅助执行动作

## 核心对象 / 核心模块

1. 任务创建工作台
2. 执行控制台
3. 报告交付中心
4. 页面内对话助手
5. 意图识别与模式路由器

## 核心流程 / 状态流 / 编排关系

### 用户主旅程

1. 用户定义任务或表达自然语言需求。
2. 系统识别任务意图并生成结构化草稿。
3. 用户确认任务范围与计划预览。
4. 系统启动研究执行图。
5. 用户在执行过程中观察进度并可进行人工干预。
6. 系统输出正式报告与诊断信息。
7. 用户选择继续修订、导出或沉淀知识。

### 对话模式切换

统一对话入口下，系统至少应长期支持以下模式：

1. `Chat Mode`
2. `Explain Mode`
3. `Task Form Mode`
4. `Task Action Mode`
5. `Research Mode`

## 关键约束与设计原则

1. 对外只有一个智能体入口，对内允许多模式执行。
2. 模式切换应由系统识别，而不是要求用户学习复杂开关。
3. 页面上下文必须参与对话理解，避免无状态问答。
4. 计划与进度展示必须结构化，便于用户理解和系统解析。
5. 关键旅程的默认体验必须优先回答“目标、状态、问题、动作、追溯”。
6. 高级诊断能力必须服务主旅程，而不是反过来挤占主旅程。

## 与其他能力域的边界

1. 本文档定义用户旅程与模式，不展开底层数据模型细节，详见 `03-domain-model-and-data-architecture.md`。
2. 本文档定义对话模式，不展开意图引擎与编排实现，详见 `06-conversation-intent-and-task-orchestration.md`。
3. 本文档定义用户在页面中的体验路径，不展开前端组件实现，详见 `07-frontend-workbench-and-delivery-center.md` 与 `12-frontend-productization-and-design-governance.md`。

## 扩展预留

1. 未来可增加长期监测型任务的持续订阅旅程。
2. 未来可增加团队协作、审批、评论和共享旅程。
3. 未来可增加移动端或消息通知触达旅程。

## 待确认问题

1. 是否需要为复杂任务引入“计划确认会话”独立模式。
2. 是否需要把“任务恢复向导”做成独立用户旅程。

## 本阶段不约束的内容

1. 不规定每条旅程必须在当前版本完整实现。
2. 不要求所有模式都必须由独立模型或独立页面承载。
