# 00 Product Overview

## 文档目的

本文档用于定义本项目的长期产品定位、核心问题、目标能力地图和长期稳定原则，是整套 `Living Blueprint SDD` 的总纲。

本文档不重复维护任务输入输出、业务规则和成功标准，这些内容统一沉淀到 `01-business-requirements.md`。

## 产品愿景与长期目标

本项目最终不是一个“输入网址后自动生成竞品分析报告”的工具，而是一个面向企业复杂研究任务的 `AI Task Operating System`。

它要服务的不是单次演示型任务，而是企业在产品、市场、战略、售前、方案和知识运营场景中持续存在的复杂研究工作。

长期目标包括：

1. 让研究任务具备标准化定义能力。
2. 让知识获取从单一网页来源升级为多源知识编排。
3. 让 Agent 从固定代码角色演进为具备上下文、工具、记忆和审计能力的运行时系统。
4. 让交付结果从“文本报告”升级为“正式报告 + 证据 + 审计 + 修订闭环”的可交付成果。
5. 让系统能够持续承载后续新增任务与需求，而不需要频繁推倒重构。

## 当前要解决的核心问题

传统竞品与研究任务通常存在以下问题：

1. 人工收集流程重复，来源分散且不完整。
2. 研究结果严重依赖个体经验，不易复用。
3. 报告结论缺乏证据链，难以审计和复核。
4. 任务一旦中断，恢复成本高。
5. 系统往往只能处理固定流程，无法适应真实任务变化。
6. 即使底层能力存在，若前端继续以调试视角组织，也无法形成真正可用的工作台。

## 产品边界

业务边界包括：

1. 系统聚焦研究任务编排、执行、审计与交付。
2. 系统不以通用聊天为目标，而是提供以研究任务为中心的对话智能体能力。
3. 系统的底层事实载体是证据与知识对象，而不是报告文本。
4. 系统的前端工作台不是附属展示层，而是研究任务操作系统的正式组成部分。

更具体的任务类型、任务输入输出和业务规则详见 `01-business-requirements.md`。

## 核心能力地图

最终产品必须具备以下能力域：

1. 任务建模与任务计划
2. 多源知识获取
3. 文档解析与知识清洗
4. RAG 检索与知识召回
5. 上下文编排
6. 短期与长期记忆
7. Agent Runtime 与任务图执行
8. 对话式统一交互
9. 意图识别与模式路由
10. 工具自主调用与权限控制
11. 质量闭环与证据审计
12. 前端工作台与交付中心
13. AI 基础设施治理

## 长期稳定原则

1. `证据优先`
   任何关键结论都应能够回指到证据来源。
2. `任务可恢复`
   长流程任务必须支持中断恢复、局部重跑和人工接管。
3. `网址非唯一入口`
   用户提供网址只是线索，不是任务成立前提。
4. `本方产品非必填`
   并非所有任务都围绕“我方产品对比竞品”。
5. `统一对话入口`
   系统对外应保持单一对话入口，对内通过意图和模式路由分流。
6. `质量闭环内生`
   初审、修订、终审应是研究链路内部能力，而不是附加动作。
7. `可审计`
   Agent 输出、工具调用、AI 供应商调用、证据引用和质量判定都应留痕。
8. `用户视角优先于技术视角`
   对用户交付的界面必须优先围绕任务目标、当前问题、建议动作和证据追溯组织，而不是围绕技术字段组织。
9. `调试能力不等于默认交付界面`
   调试、链路、原始配置和原始结构体可以存在，但不应默认污染正式工作台。
10. `产品化工作台是核心能力，不是外层包装`
   如果工作台不可理解、不可操作、不可审计，再强的底层能力也无法构成合格产品。

## 当前结论

1. 当前工程是蓝图的起点，而不是蓝图本身。
2. 当前 `多 Agent + DAG + 报告生成` 形态仍偏向可演示工作流系统。
3. 最终产品应围绕“研究任务操作系统”而不是“报告生成器”演进。
4. 从当前基线继续前进时，必须同时治理工作台体验与底层骨架，不能再允许“后端越来越强，前端继续堆叠调试面板”的分裂演化。

## 扩展预留

1. 未来可扩展到文档上传、连接器接入、组织级知识平台。
2. 未来可引入协作权限、团队空间、知识运营看板等企业能力。
3. 未来可扩展到更多研究任务模板与行业方法论。

## 待确认问题

1. 长期是否需要把“组织级协同工作台”单独提升为一级产品形态。
2. 是否需要把“外部连接器生态”视为独立产品能力线。

## 本阶段不约束的内容

1. 不要求当前实现已经具备全部能力域。
2. 不要求当前路线图一次性覆盖长期全部交付物。
