# 06 Conversation Intent And Task Orchestration

## 文档目的

本文档用于定义系统如何通过统一对话入口承接聊天、解释、填表、动作与研究执行请求，并把自然语言请求转换成系统内部可执行的模式与动作。

## 本能力域在系统中的职责

本能力域负责：

1. 识别用户意图。
2. 将请求路由到合适模式。
3. 在对话与任务系统之间建立正式桥梁。

## 业务目标与用户价值

1. 让用户不需要学习复杂入口和控制面板。
2. 让自然语言既能解释系统，也能驱动任务。
3. 让系统始终知道自己处于哪个任务、哪个页面和哪种能力边界内。

## 当前结论

系统对外应保持单一对话入口，对内至少拆分五种模式：

1. `Chat Mode`
2. `Explain Mode`
3. `Task Form Mode`
4. `Task Action Mode`
5. `Research Mode`

## 目标能力

1. 基于上下文的意图识别
2. 基于模式的执行分流
3. 自然语言填表
4. 自然语言触发任务动作
5. 智能体自我认知与边界表达

## 关键场景

1. 用户说“介绍一下这个系统”，系统进入 `Chat / Explain Mode`。
2. 用户说“帮我做一个飞书和 Notion 的竞品分析”，系统进入 `Task Form Mode`。
3. 用户说“把重点改成 AI 能力和定价”，系统更新表单草稿。
4. 用户说“从 rewrite_report 开始重跑”，系统进入 `Task Action Mode`。
5. 用户说“继续补搜 pricing 证据”，系统进入 `Research Mode` 或动作增强链路。

## 核心对象 / 核心模块

1. `Conversation Agent`
2. `Intent Recognition Engine`
3. `Mode Router`
4. `Form Draft Builder`
5. `Task Action Translator`
6. `Agent Identity Layer`

## 核心流程 / 状态流 / 编排关系

1. 用户输入自然语言。
2. 系统构建页面、任务、节点、历史会话等上下文。
3. `Intent Recognition` 判定当前意图。
4. `Mode Router` 选择执行模式。
5. 系统生成解释内容、结构化表单字段、任务动作命令或研究任务请求。
6. 执行结果写回对话上下文、草稿对象和审计对象。

## 关键约束与设计原则

1. `一个入口，多种内部模式`
   用户不应感知系统内部复杂链路。
2. `意图决策必须可审计`
   为何把一句话识别成某类动作，应可以追溯。
3. `对话必须有系统自我认知`
   智能体要知道自己是谁、当前在哪、能做什么、不能做什么。
4. `对话不应绕开系统边界`
   对话产生的动作必须经过权限、校验和任务编排。

2. 不要求当前阶段完成所有动作类型的自然语言控制。
