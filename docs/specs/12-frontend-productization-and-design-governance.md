# 12 Frontend Productization And Design Governance

## 文档目的

本文档用于定义本系统所有用户可见前端界面的产品化治理规则、设计落地约束和验收门槛，避免工作台继续退化为“开发调试面板拼装层”。

本文档不负责定义页面终态信息架构本身，这部分由 `07-frontend-workbench-and-delivery-center.md` 承担；本文档负责定义“这些页面必须按什么规则被实现、评审和验收”。

## 本能力域在系统中的职责

本能力域负责：

1. 把“用户视角优先于技术视角”固化为强制执行规则。
2. 把设计系统、组件拆分、复用抽象和调试能力收纳从“建议”提升为“验收门槛”。
3. 为 `Phase 3` 及之后所有前端改造提供统一治理基线。

## 业务目标与用户价值

1. 降低用户理解任务、问题和动作的认知负担。
2. 避免技术术语、原始 JSON 和调试结构体直接污染交付界面。
3. 让前端能够长期稳定演进，而不是每个阶段都再做一次“大文件重写”。

## 当前结论

1. `specs/07` 定义工作台的信息架构与用户体验目标。
2. 本文档定义这些目标在实现层必须满足的强制约束。
3. 从 `Phase 3` 开始，所有新建或重构的用户界面都必须遵守本文件。
4. `Phase 1`、`Phase 2` 已完成页面不要求为了追求叙事一致而回溯重写，但后续只要触碰相关页面，就必须按本文件收口。

## 适用范围

本文件适用于以下对象：

1. 任务创建工作台、执行控制台、报告交付中心及其子组件。
2. 所有用户可见的状态文案、错误提示、诊断视图、来源视图和证据视图。
3. 与 `SSE`、任务状态、节点状态、质量诊断、证据追溯相关的公共前端能力。

## 目标能力

1. 真正落地的设计 Token 体系
2. 面向用户目标的问题中心式信息架构实现
3. 可维护的组件拆分与公共能力复用
4. 技术术语向用户语言的稳定翻译层
5. 受控的调试与高级诊断展示机制
6. 可审计的设计评审、代码评审与验收门槛

## 关键约束与设计原则

1. `用户视角优先`
   页面首先回答用户“我要完成什么、系统现在怎么了、我接下来该做什么”，而不是先暴露内部实现结构。
2. `问题处理优先于字段堆叠`
   页面应围绕目标、状态、问题、动作和追溯组织，而不是围绕接口字段和调试数据组织。
3. `渐进披露`
   默认界面只展示完成当前任务所需的关键信息，原始结构和底层链路必须后置。
4. `一致性优先于局部快修`
   任何局部功能都不能以牺牲全局文案、状态提示、样式体系和复用边界为代价。

## 强制治理规则

### 1. 用户语言与术语规则

1. 用户主界面的一级标题、一级卡片标题、主操作文案不得直接使用纯技术术语，例如 `DAG`、`SSE`、`fallback`、`raw config`、`raw output`、`Last-Event-ID`。
2. 内部术语若必须出现在界面上，必须先翻译为用户可理解语言，再在高级说明中补充技术名词。
3. 页面标题必须描述用户目标、当前问题或下一步动作，不能只描述内部机制。
4. `QualityCheckpoint`、`RevisionPlan`、`NodeTrace` 等内部对象名不得直接作为用户一级体验语言，必须转译为“质量检查结果”“修订建议”“节点追踪”等业务可读表达。

### 2. 信息架构与披露规则

1. 任一页面首屏必须优先展示以下内容中的大部分：任务目标、当前阶段、关键问题、建议动作、结果摘要。
2. 原始 JSON、原始配置、原始输入输出、事件游标、技术 trace 等信息不得作为首屏默认内容。
3. 任何高级诊断信息都必须通过“展开”“高级诊断”“调试模式”等显式交互进入，不允许默认展开。
4. 任何原始技术块出现前，必须先给出业务可读摘要，说明这段信息为什么对用户有意义。

### 3. 组件与文件拆分规则

1. 用户界面组件和页面文件应以单一职责为原则组织。
2. 页面组件或复杂业务组件建议控制在 `300` 行以内；超过 `300` 行时必须评估拆分；超过 `500` 行时必须在评审中给出书面豁免理由。
3. 不允许在页面文件顶部堆积大段工具函数、解析函数和状态映射逻辑；此类逻辑应下沉到公共工具模块或专用 hooks。
4. 同一语义的解析函数、文案映射函数、状态提示函数、错误提示函数，不允许在两个及以上文件中重复实现而不抽取公共层。

### 4. 设计系统落地规则

1. 用户界面中的颜色、字号、间距、圆角、阴影、状态色必须优先消费统一 Token，而不是散落硬编码。
2. 若新增用户可见样式值，必须同步判断其是否应进入 Token 体系；不能出现“Token 已定义但页面完全不引用”的死定义。
3. 状态色语义必须统一：
   - 红色仅用于失败、阻塞、严重错误
   - 橙色用于警告、待处理、暂停
   - 绿色用于成功、通过
   - 蓝色用于进行中、激活中
   - 灰色用于说明、技术属性和次要状态
4. 任一阶段验收前，若用户界面仍大量依赖硬编码颜色和一次性状态样式，视为设计系统未落地完成。

### 5. 交互与状态反馈规则

1. 加载中、空状态、错误状态、重连中、兜底模式等反馈必须使用统一提示模式。
2. `SSE` 断线、重连、兜底轮询等提示不得在多个页面各自实现一套文案和交互。
3. 问题提示必须尽量靠近对应动作出现，例如“缺证据章节”旁应直接出现“查看证据”“前往补证”“触发重写”等动作，而不是只列问题。
4. 人工干预入口必须围绕问题场景呈现，不能要求用户先理解技术状态机再决定操作。

### 6. 调试与高级诊断规则

1. 调试面板不是正式产品界面的默认组成部分。
2. `Debug`、原始配置、原始输入输出、链路 trace、事件回放明细只能存在于以下两类区域：
   - 明确命名的“高级诊断”视图
   - 显式开启的调试模式
3. 生产验收不能依赖“让用户自己去看原始 JSON 才能理解问题”的路径。
4. 任一页面若需要暴露高级诊断，也必须先提供摘要结论、问题归因和建议动作。

### 7. 公共能力复用规则

1. 以下能力必须优先抽取为共享层，而不是页面内重复实现：
   - JSON 解析与安全降级
   - 状态文案映射
   - 流式事件连接状态提示
   - 质量问题分类与展示
   - 证据来源标签与来源说明展示
2. 若在两个以上组件中发现重复逻辑但仍未抽取公共能力，则该改动不应视为阶段验收完成。

### 8. 审查与验收门槛

1. 从 `Phase 3` 开始，所有用户界面重构必须经过至少一次信息架构与文案审查。
2. 代码评审必须显式检查以下内容：
   - 是否继续暴露技术术语作为主界面语言
   - 是否存在默认展开的原始调试信息
   - 是否继续新增重复工具函数、重复状态映射或重复错误提示实现
   - 是否违反组件复杂度和拆分规则
3. 阶段验收必须包含至少一次“非实现者可理解性检查”：
   - 由未参与该页面实现的人进行
   - 核心问题是“他是否能在不阅读代码的情况下理解当前任务、问题和下一步动作”
4. 若页面只能由开发者解释后才能使用，则该页面不应被判定为完成产品化验收。

### 9. 例外机制

1. 允许存在短期技术债，但必须在对应阶段计划、任务拆解或评审记录中明确写出原因和收口时点。
2. 以下规则原则上不允许豁免：
   - 技术术语不得直接占据主界面
   - 调试信息不得默认暴露
   - 关键状态与动作不得缺少用户可读解释

## 与其他能力域的边界

1. 本文档不定义具体页面终态布局，详见 `07-frontend-workbench-and-delivery-center.md`。
2. 本文档不定义质量规则本身，详见 `09-quality-evidence-and-audit.md`；但约束这些质量对象如何被产品化呈现。
3. 本文档不定义阶段路线，详见 `10-phased-roadmap.md`；但从 `Phase 3` 开始，路线图中的前端建设必须遵守本文件。

## 阶段映射关系

1. `Phase 1-2`
   作为历史基线存在，不回溯改写其既成事实，但后续只要触碰相关页面，就应按本文件收口。
2. `Phase 3`
   作为本文件第一次完整落地的阶段，必须完成前端产品化治理基线建立。
3. `Phase 4-5`
   所有新增能力都必须长在已经治理过的工作台骨架上，而不是重新长出新的调试面板。

## 扩展预留

1. 后续可继续拆出更细的视觉设计系统、内容设计规范和多租户控制台设计规范。
2. 后续可将部分规则沉淀为自动化检查，例如样式 Token 使用校验、页面复杂度提示和术语扫描。

## 待确认问题

1. 是否需要把部分规则进一步转化为 ESLint、Stylelint 或构建时检查。
2. 是否需要单独维护一份“用户语言术语表”，用于统一技术术语转译。

## 本阶段不约束的内容

1. 不规定必须采用哪一个具体图表库、组件库或视觉风格实现。
2. 不要求一次性清空全部历史前端技术债，但要求 `Phase 3` 及之后的重构页面不得继续制造同类问题。
