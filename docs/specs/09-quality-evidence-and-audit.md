# 09 Quality Evidence And Audit

## 文档目的

本文档用于定义系统中的证据约束、质量闭环、修订计划和审计体系，确保研究结果既可交付也可复核。

## 本能力域在系统中的职责

本能力域负责：

1. 定义什么叫“有证据支撑的结论”。
2. 定义质量检查如何驱动修订与补证。
3. 定义执行、工具和 AI 调用如何被审计。

## 业务目标与用户价值

1. 减少无依据结论和空泛报告。
2. 让质量问题转化为可执行修订动作。
3. 让用户和组织能追溯任务全过程。

## 当前结论

最终系统必须坚持以下核心判断：

1. 证据是系统的基础事实层。
2. `QualityCheckpoint` 是内部过程产物，不是面向交付的独立报告。
3. 初审失败后应输出质量结果和修订计划，而不是再生成一篇“评审报告”。
4. 任何关键结论都应具备可回指证据。
5. 质量结果必须以“帮助用户处理问题”的方式被消费，而不是以“展示原始评审结构体”的方式被消费。

## 目标能力

1. 字段级证据约束
2. 章节级证据约束
3. 结论级证据约束
4. 质量 checkpoint 与问题分级
5. 修订计划驱动重写与补证
6. 执行轨迹、工具轨迹、AI 调用审计

## 关键场景

1. 抽取阶段记录字段级 `sourceUrls` 与 `evidenceIds`。
2. Reviewer 识别关键章节缺证据。
3. 系统判断问题是应先补证还是直接重写。
4. 用户追溯报告中的某条结论来自哪些来源。
5. 用户查看某个节点为何失败、调用了哪些模型和工具。
6. 用户在报告交付中心先看到结论、问题和动作，而不是先看到原始质量结构。

## 核心对象 / 核心模块

1. `Evidence Coverage Rules`
2. `QualityCheckpoint`
3. `RevisionPlan`
4. `Review Next Actions`
5. `Audit Engine`
6. `AgentExecutionLog`
7. `ToolExecutionTrace`
8. `AIAuditLogger`

## 核心流程 / 状态流 / 编排关系

1. 采集和抽取阶段生成证据与知识对象。
2. 分析和写作阶段消费证据并写出引用关系。
3. 评审阶段根据字段、章节和结论级规则生成 `QualityCheckpoint`。
4. 若存在问题，系统输出 `RevisionPlan` 与 `Next Actions`。
5. 编排系统根据问题类型选择补证、重写或人工接管。
6. 整个过程的 Agent、工具与 AI 调用都进入审计链。

## 关键约束与设计原则

1. `无幻觉原则`
   数据 schema 中必须长期包含可回指来源字段。
2. `证据覆盖优先于文本完整`
   一份结构漂亮但无证据的报告不视为高质量。
3. `质量结果必须可执行`
   不能只给分或只列问题，要给出后续动作。
4. `审计数据必须可关联`
   任务、节点、Agent、工具、模型供应商之间的轨迹要能串起来。
5. `质量结果必须产品化呈现`
   用户默认看到的应该是问题摘要、影响范围、建议动作和关键追溯入口，而不是原始技术对象。
6. `原始质量结构不得默认暴露`
   原始评分因子、原始评审结构体和深层审计明细属于高级诊断信息，应后置到受控视图。

## 交付体验层约束

从用户消费质量结果的角度，必须长期遵守以下约束：

1. `QualityCheckpoint` 不应以“技术诊断报告”形式直接对外裸露。
2. `RevisionPlan` 必须转化为用户可理解的动作建议，例如补证、重写、复核、回到任务页处理。
3. 报告页中的质量结果默认顺序应优先回答：
   - 当前是否通过
   - 最大问题是什么
   - 用户下一步该做什么
   - 如果要追溯依据，去哪里看
4. 原始质量 JSON、知识覆盖 JSON、深层评分细节只能作为高级诊断信息存在。

## 与其他能力域的边界

1. 本文档不定义完整页面信息架构，详见 `07-frontend-workbench-and-delivery-center.md`。
2. 本文档定义质量对象在交付体验层的消费原则；前端治理和展示门槛详见 `12-frontend-productization-and-design-governance.md`。
3. 本文档不定义任务图执行框架，详见 `04-agent-runtime-and-workflow-architecture.md`。
4. 本文档不替代数据模型定义，详见 `03-domain-model-and-data-architecture.md`。

## 扩展预留

1. 后续可引入更细的质量评分体系和组织级评审标准。
2. 后续可引入质量趋势分析、成本质量关联分析和自动告警。
3. 后续可支持不同任务模板下的差异化证据规则。

## 待确认问题

2. 是否需要对不同角色开放不同粒度的审计视图。

## 本阶段不约束的内容

1. 不强制要求当前所有章节都具备完全自动化证据校验。
2. 不规定当前必须上线完整质量评分看板。
