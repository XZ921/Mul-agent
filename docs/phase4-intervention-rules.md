# Phase 4 干预状态规则

## 目标

Phase 4 把任务编排正式收口为异步事件链路，因此任务详情、恢复入口和说明文档必须共享同一套状态语义，避免不同界面对同一个节点给出不一致解释。

## 节点状态机

- `WAITING_RETRY`：节点执行失败，但仍处于系统允许的自动重试窗口。前端需要把它解释为“系统还会继续尝试”，而不是立即要求人工接管。
- `WAITING_INTERVENTION`：节点已经耗尽自动恢复能力，需要人工确认下一步动作。任务详情页要明确展示阻塞原因、候选动作和影响范围。
- `COMPENSATED`：节点不再继续原动作，而是通过补偿逻辑完成状态收口。这个状态强调“系统已做过一次可追踪恢复”，而不是简单跳过。
- `DLQ`：节点事件多次失败后进入死信轨迹。进入 `DLQ` 后，系统必须保留失败摘要、最近错误和恢复建议，供人工排障或二次编排使用。

## 恢复与可视化约束

- 任务恢复必须围绕“保留成功检查点、只重放必要节点”展开，不能把已经成功的证据采集和报告产物全部清空。
- 人工恢复时要同步展示当前阶段、已完成步骤、剩余步骤和步骤状态，保证任务计划与进度可追溯。
- 任务详情页里的人工干预说明，必须与系统 prompt 中的阶段输出规范保持一致。

## 动态补图与检索边界

- 当终审发现关键证据仍有缺口时，可以进入**动态补图**分支，把额外的采集、复核和报告修订节点挂接到当前计划版本上。
- 动态补图新增的节点仍然要延续同一套状态机，不能绕开 `WAITING_INTERVENTION`、`COMPENSATED` 或 `DLQ` 语义。
- 动态补图触发后，新补回的证据仍然要携带 `sourceUrls`，保证报告结论和补充证据可溯源。

## Task RAG、对话与动作边界

- `Task RAG` 的职责是围绕当前任务证据、检索片段和运行状态回答“为什么停在这里”“还缺什么证据”等解释型问题。
- 对话入口只能基于当前任务上下文解释状态、证据和恢复建议，不能越权承诺超出任务边界的执行结果。
- 当用户想继续推进任务时，对话层只能给出**动作预览**，例如“从某个节点重跑”“继续补充官方文档证据”，并要求明确确认后再执行。
- 动作预览必须绑定目标节点、动作类型、是否需要确认和关联 `sourceUrls`，避免解释层与执行层脱节。
