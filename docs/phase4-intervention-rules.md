# Phase 4 Intervention Rules

## Runtime States

Phase 4 明确引入以下运行态，用来约束恢复、补偿与人工介入边界：

- `WAITING_RETRY`：命中可重试的瞬时失败，系统仍会在预算内自动重试。
- `WAITING_INTERVENTION`：自动重试预算耗尽，或失败已被判定为必须人工介入。
- `COMPENSATED`：节点已经通过补偿策略安全收口，但不会伪装成正常成功。
- `DLQ`：进入死信留痕，保留失败摘要、重试轨迹与可追溯上下文，便于后续审计与恢复。

## Recovery Boundary

Phase 4 的恢复链路不是无限自动化，而是围绕“先保证可追溯，再决定是否继续”设计：

- 自动恢复只覆盖 `WAITING_RETRY` 之前的预算内重试。
- 当节点进入 `WAITING_INTERVENTION` 或需要 `COMPENSATED` 收口时，系统必须把当前状态暴露给人工操作入口。
- `DLQ` 只负责沉淀失败事实，不负责暗中替用户继续推进任务。

## Conversation Boundary

这一阶段已经把动态补图、Task RAG、对话与动作预览收进统一边界，但仍保持克制：

- 动态补图只允许基于已有计划版本和现有证据缺口生成补救动作。
- Task RAG 负责解释“当前为什么卡住、缺什么证据、有哪些来源”，不直接替代执行链路。
- 对话入口负责解释、澄清与引导，不把内部事件流原样暴露成首屏术语。
- 动作预览只展示可执行动作、影响范围与风险提示，避免把高风险动作伪装成普通问答。

## Phase Note

Phase 4 的目标是把干预状态机、解释入口和证据回指说清楚，而不是提前承诺更高阶段的自治执行能力。
