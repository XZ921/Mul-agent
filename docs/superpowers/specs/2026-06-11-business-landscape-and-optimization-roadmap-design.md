# AI 竞品分析 Agent 协作系统业务全景与功能优化路线图设计

## 总览层

### 文档目的

本文档用于给当前工程建立一张可持续演进的总控母图，避免后续继续以“哪个类先坏了就修哪个类”的方式碎片化推进。

本文档回答四个核心问题：

1. 当前系统完整业务全景到底由哪些主业务链路组成。
2. 每条链路分别由哪些执行引擎承接。
3. 每条链路与平台底座分别应该如何优化。
4. 后续继续补写 `ExtractorAgent.md`、`WriterAgent.md`、`ReviewerAgent.md` 等专题文档时，有哪些全局红线不应重复发明。

### 当前结论

1. 当前系统不应再被理解为“若干 Agent 的堆叠”，而应被理解为：
   `9 条主业务链路 + 4 个业务执行引擎 + 6 个平台底座`。
2. 业务链路、执行引擎、平台底座三者不是对等关系，而是：
   `业务链路 > 执行引擎 > 平台底座`。
3. `搜索与采集` 链路已经形成了高密度问题样板，`[CollectorAgent.md](/e:/java_study/Mul-agnet/docs/problem/CollectorAgent.md)` 应作为本设计中的第一份深度专题基线，而不是被重新发明。
4. 后续优化的正确顺序不是“先平台、后业务”，而是：
   `先收敛业务链路语义 -> 再收敛执行引擎边界 -> 最后下沉到底座治理`。
5. 本文档短期服务于“竞品分析平台”主航道，长期预留为“通用研究任务操作系统”能力骨架。

## 1. 业务与功能全景图 (Business & Feature Map)

### 1.1 主业务链路总图

当前系统的主战场应被拆成以下 9 条业务链路：

1. 任务定义与编排
2. 搜索与采集
3. 提取结构化
4. 分析推理
5. 报告写作
6. 质量审查
7. 修订与重写
8. 对话协同
9. 交付与审计

### 1.2 主业务链路表

| 业务板块名称 | 核心功能组件 | 业务痛点 / 潜在瓶颈 | 与执行引擎的映射关系 |
| --- | --- | --- | --- |
| 任务定义与编排 | `CreateTaskRequest`、`TaskDefinitionAppService`、`WorkflowFactory`、`WorkflowPlan`、`SearchExecutionPlan`、计划预览 | 任务输入模型仍偏早期竞品分析器；任务定义、计划生成、回退策略、阶段规划语义被拆散到配置层与平台层；预览态与运行态契约未完全对齐 | `任务执行引擎` |
| 搜索与采集 | `CollectorAgent`、`SearchExecutionCoordinator`、`CandidateVerifier`、`CollectionTargetSelector`、`BrowserSearchRuntimeService`、`SourceDiscoveryService`、`SourceCollector` | 搜索补源质量波动大；动态抓取与反爬对业务结果影响极大；候选验证、正式采集、搜索现场审计和恢复语义存在断层 | `搜索执行引擎` |
| 提取结构化 | `SchemaExtractorAgent`、`SchemaService`、`EvidenceFragment`、`SectionEvidenceBundle`、字段覆盖模型 | 抽取结果高度依赖上游证据纯度；字段级 `sourceUrls`、`evidenceIds`、结构化 Schema 仍有漂移风险；缺少更明确的抽取契约层 | `任务执行引擎` |
| 分析推理 | `CompetitorAnalysisAgent`、`AnalysisResult`、分析覆盖对象、知识草稿 | 结论质量受提取质量强依赖；分析层容易把“证据不足”错误包装成“分析结论”；分析输出和写作输入的共享契约有继续膨胀风险 | `任务执行引擎` |
| 报告写作 | `ReportWriterAgent`、`Report`、章节证据束、报告正文和摘要 | 报告写作与交付聚合边界交错；正文、引用、问题标记、复用理由散落在大对象中；写作阶段的稳定输入协议仍需收口 | `任务执行引擎` |
| 质量审查 | `QualityReviewAgent`、`QualityCheckpoint`、`QualityDiagnosis`、`RevisionPlan`、`ReportDiagnosisAssembler` | 质量结果既承担内部编排语义，又承担外部交付解释语义；缺少更稳定的问题分类和动作映射层；原始诊断结构仍然过重 | `质量回流引擎` |
| 修订与重写 | `CompensationGraphAssembler`、`DynamicTaskGraphService`、`TaskRuntimeCommandAppService`、`RecoveryEngine`、`RevisionDirective` | 当前是最容易被低估的一条链路；补证、重写、局部重跑、人工接管都在这里汇合，但尚未被提升为独立业务功能 | `质量回流引擎` |
| 对话协同 | `ConversationService`、`IntentRecognitionService`、`ModeRouter`、`TaskActionTranslator`、`ConversationAgent` | 统一入口已经形成，但模式判断、动作预览、正式执行、解释语义仍纠缠在同一服务；对话链路容易越过正式业务边界 | `对话动作引擎` |
| 交付与审计 | `ReportService`、`ExportPackageService`、`ReportExportRenderer`、`TaskEventReplayService`、`TaskSseHub`、`AIAuditLogger` | 交付结果、证据追溯、质量摘要、回放与审计聚合在少数大类中；实时观察与历史回放、主路径与高级诊断边界尚不清晰 | `任务执行引擎` + `质量回流引擎` |

### 1.3 业务执行引擎表

| 执行引擎 | 主要承接对象 | 当前痛点 / 潜在瓶颈 |
| --- | --- | --- |
| 任务执行引擎 | DAG、节点状态、动态补图、重跑、恢复、人工接管、任务推进 | 当前大量语义集中在 `DagExecutor`、`WorkflowFactory`、`TaskRuntimeCommandAppService`；技术节点语义和业务阶段语义尚未彻底分层 |
| 搜索执行引擎 | 搜索计划、候选生成、候选验证、补源顺序、正式选源、搜索现场回放 | 搜索计划和运行时行为已经明显是业务规则，但仍常被当作平台杂项；恢复、缓存、回放、观察的契约未完全稳定 |
| 质量回流引擎 | 质量问题 -> 补证 / 重写 / 重跑 / 人工接管 的动作转换 | `质量审查` 与 `修订与重写` 之间的桥梁已经存在，但缺少独立引擎视角；质量结果驱动后续图的规则仍分散 |
| 对话动作引擎 | 意图识别、模式路由、动作预览、确认执行、任务控制提交 | 对话目前更像“大服务聚合层”，缺少动作执行协议、风险确认协议和正式命令协议的清晰边界 |

### 1.4 平台底座表

| 平台底座 | 核心功能组件 | 业务痛点 / 潜在瓶颈 |
| --- | --- | --- |
| 知识摄取 / RAG / 记忆 | `KnowledgeIngestionService`、`TaskRetrievalService`、`TaskRetrievalIndexService`、`MemoryFusionService`、`MemoryWritebackService` | 资料接入、检索、记忆、复用都在演进，但任务级 / 领域级 / 组织级边界仍需要更正式的契约 |
| AI 与工具治理 | `ModelGateway`、`ProviderRegistry`、`RoutingPolicy`、`BudgetGuard`、`PromptTemplateService`、`AIAuditLogger` | 已经有统一网关雏形，但策略治理、预算门槛、工具权限和降级语义仍需继续收口 |
| 可观测 | `TaskEventPublisher`、`TaskSseHub`、`AgentLogService`、任务快照、搜索进度事件 | 实时观察能力已经存在，但任务级视角、节点级视角、搜索现场视角还未完全统一 |
| 恢复与回放 | `TaskRecoveryService`、`RecoveryEngine`、`TaskEventReplayService`、`TaskSnapshotCacheService`、检查点对象 | 当前更偏“恢复语义存在”，还没完全做到“恢复现场可解释”；搜索现场和任务现场的恢复粒度不一致 |
| 质量 / 配额 / 安全治理 | `OrganizationQuotaPolicy`、`TaskQuotaCoordinator`、`UrlSecurityUtils`、`HttpUrlOnlyValidator`、质量规则对象 | 质量治理、预算治理、安全治理虽然可合并讨论，但职责 owner 长期不应混在一个心智模型里 |
| 多渠道资料接入 | `KnowledgeController`、`KnowledgeIngestionRequest`、`ConnectorSyncService`、`ConnectorDefinition` | 当前能力已经显露入口，但“上传资料 / API 资料 / 认证资料 / 未来连接器”尚未被提升成正式产品能力线 |

## 2. 分模块优化方案 (Modular Optimization Deep Dive)

### 2.1 主业务链路优化

#### 2.1.1 任务定义与编排

- 优化目标：
  让“定义任务”和“决定任务如何运行”回到同一条正式业务链路，避免任务表单、计划预览、工作流工厂、搜索计划、重跑语义彼此割裂。
- 核心技术栈 / 方案：
  - 任务契约分层：`Task Draft` / `Task Definition` / `Execution Plan`
  - 计划语义对象：阶段、分支、回退策略、目标约束
  - 计划预览与运行态分离 DTO
  - `WorkflowFactory` 业务化拆分：计划生成器、阶段规划器、节点模板装配器
- 执行路径：
  - 止血：
    - 对齐 `CreateTaskRequest` 与业务蓝图字段，至少补齐可选 `subjectProduct`、质量要求、预算和补充资料预留。
    - 把任务预览接口和运行态节点接口的正式展示差异写成契约说明，避免前端继续混用。
    - 为 `WorkflowFactory` 中的搜索计划构建、节点命名、依赖拼装加更明确的单元测试。
  - 治本：
    - 引入正式的 `TaskDefinitionAggregate` 和 `ExecutionPlanAggregate`，把任务输入、计划预览、运行计划从 JSON 字符串拼接中拉出来。
    - 将 `WorkflowFactory` 拆成“业务计划器”和“技术装配器”两层。
    - 为“任务定义与编排”建立独立专题文档，作为后续其它链路专题的母线。
  - 提效：
    - 为计划预览引入缓存与版本摘要，降低频繁预览成本。
    - 将计划生成中的高频只读数据查询做批量化。
    - 在任务创建阶段引入预算闸门和计划复杂度评分，提前阻断无效大任务。

#### 2.1.2 搜索与采集

- 优化目标：
  让搜索补源、候选验证、动态抓取、反爬、正式采集、来源去噪形成一条稳定、可恢复、可回放、可解释的业务链路。
- 核心技术栈 / 方案：
  - 搜索执行计划与搜索现场审计双对象
  - 候选验证器、选源器、浏览器补源器、正式采集器分治
  - 反爬与动态抓取策略库
  - 一次采集快照复用、搜索现场断点恢复、正式回放
  - `CollectorAgent.md` 作为链路专题基线
- 执行路径：
  - 止血：
    - 直接吸收 `[CollectorAgent.md](/e:/java_study/Mul-agnet/docs/problem/CollectorAgent.md)` 中已定位的问题，优先修正：
      - `OFFICIAL` 免检放行
      - 中文特征词缺失
      - 候选排序被广告高权威分绑架
      - 验证期与正式采集期双重爬取
      - 搜索事件和回放信息缺损
    - 调整默认搜索引擎、兜底顺序和候选阈值配置，先把搜索噪声压下去。
    - 补齐 `searchAudit`、`selectedTargets`、搜索现场回放所需字段的最小事件契约。
  - 治本：
    - 把 `SearchExecutionCoordinator`、`CollectionTargetSelector`、`CandidateVerifier`、`BrowserSearchRuntimeService` 收敛成独立的“搜索执行引擎”子域。
    - 将 `SearchExecutionPlan`、`SearchAuditSnapshot`、`SearchExecutionTrace` 从“半结构化附属对象”升级为正式业务契约。
    - 建立 `CollectorAgent` 链路的 Facade 和专题基线，后续同类问题不再散落在 `ReportService`、`TaskEventReplayService` 等外围服务里修补。
  - 提效：
    - 引入验证快照复用、浏览器结果页轻量预览、正式采集快照优先复用，降低公网撞击次数。
    - 对搜索计划生成、候选去重、候选排序引入批量计算和缓存。
    - 增加预算闸门、反爬熔断、Provider 降级轨迹记录，形成稳定的成本和风控面板。

#### 2.1.3 提取结构化

- 优化目标：
  让抽取阶段成为“把原始证据转成可机读正式知识对象”的稳定环节，而不是继续依赖模型自行发挥输出字段结构。
- 核心技术栈 / 方案：
  - `SchemaExtractorAgent` 契约收口
  - 字段级覆盖规则对象
  - `EvidenceFragment` / `SectionEvidenceBundle` 正式化
  - 抽取结果 Golden Master
- 执行路径：
  - 止血：
    - 为关键字段缺 `sourceUrls`、缺 `evidenceIds` 的情况补兜底回填。
    - 对已有 Schema 的空值、缺字段、字段别名漂移加兼容测试。
    - 把抽取输出中的核心结构固定为：字段值、字段证据、字段状态、章节证据摘要。
  - 治本：
    - 将 `SchemaExtractorAgent` 中的“Prompt 构造 / 模型输出修正 / 证据回填 / 覆盖计算”拆成小型协作对象。
    - 把抽取链路的共享契约独立成专题文档，后续分析和写作只能消费这一层稳定对象。
    - 为抽取输出建立 Golden Master 快照测试，防止后续字段语义静默漂移。
  - 提效：
    - 对重复证据引用建立去重和共享索引。
    - 对高频 Schema 引入模板缓存和字段规则缓存。
    - 对跨竞品抽取的相同字段走批量校验与合并统计。

#### 2.1.4 分析推理

- 优化目标：
  让分析阶段输出“有证据边界的判断”，而不是把上游信息缺口在语言层掩盖掉。
- 核心技术栈 / 方案：
  - 分析输入标准化对象
  - 结论级证据绑定
  - 竞争优势 / 风险 / 缺口 三类判断模型
  - 分析输出 Golden Master
- 执行路径：
  - 止血：
    - 在分析输出缺 `sourceUrls` 或证据不足时，显式输出“证据不足”标记。
    - 为结论与字段覆盖不一致的情况补防御性测试。
    - 收口分析结果的最小共享契约，避免写作阶段继续猜字段。
  - 治本：
    - 将 `CompetitorAnalysisAgent` 中的“输入整理 / 对比推理 / 结果规范化 / 证据回填”拆层。
    - 引入分析规则对象，把“结论类型”和“证据最低要求”正式建模。
    - 建立分析链路专题文档，明确它不负责补证，只负责在边界内推理。
  - 提效：
    - 对重复竞品比较模式做模板缓存。
    - 将高频统计和对比逻辑做结构化复用，减少纯模型重复推理。
    - 引入分析预算和超时闸门，避免大任务长尾。

#### 2.1.5 报告写作

- 优化目标：
  让写作阶段只负责“把结构化分析结果转成正式交付文稿”，不再承担过多补证、修复、兼容、聚合职责。
- 核心技术栈 / 方案：
  - 写作输入 DTO 固化
  - 章节模板层
  - 引用落稿规范
  - 报告正文与交付聚合分离
- 执行路径：
  - 止血：
    - 固定写作阶段输入至少包含：分析内容、证据片段、问题标记、章节证据束。
    - 为“报告正文生成成功但引用缺失”的情况补阻断规则。
    - 缩小 `ReportWriterAgent` 兼容旧格式的范围，先保证新格式稳定。
  - 治本：
    - 将 `ReportWriterAgent` 的“输入标准化 / Prompt / 正文生成 / 结果落库 / 记忆写回”拆开。
    - 将写作阶段与 `ReportService` 的读取聚合边界分开，避免写作输出继续被交付聚合层反向污染。
    - 建立报告写作专题文档，统一章节模板和引用协议。
  - 提效：
    - 对章节模板和报告骨架做缓存。
    - 对正式写作与轻量改写走不同模型和预算策略。
    - 对重复报告读取引入投影缓存。

#### 2.1.6 质量审查

- 优化目标：
  让质检阶段稳定回答三件事：当前是否可交付、最大问题是什么、下一步动作是什么。
- 核心技术栈 / 方案：
  - `QualityCheckpoint` / `RevisionPlan` / `NextAction` 三分层
  - 问题分类规则
  - 质量产品化视图
- 执行路径：
  - 止血：
    - 为缺证据、结构缺失、表达问题三类典型问题补稳定分类。
    - 把 Reviewer 输出中仍靠文本描述承载的关键动作，补成结构化字段。
    - 为“质检通过但仍有高风险诊断”的情况补回归测试。
  - 治本：
    - 将 `QualityReviewAgent` 拆成“评估规则层 + 诊断聚合层 + 动作建议层”。
    - 为质量问题建立正式代码表，供交付页、回流引擎、对话动作共用。
    - 把质量结果与交付体验之间的映射单独做成组装器，不再直接暴露原始审查对象。
  - 提效：
    - 对质量计算中高频读取的报告证据、章节覆盖做缓存。
    - 批量化查询证据、知识、节点执行信息。
    - 对稳定通过的轻量复核链路使用更小成本模型。

#### 2.1.7 修订与重写

- 优化目标：
  让“补证、重写、局部重跑、人工接管、恢复收口”从隐含流程升级为正式业务链路。
- 核心技术栈 / 方案：
  - 动态补图模型
  - 受影响节点分析
  - 回流动作协议
  - 恢复检查点正式化
- 执行路径：
  - 止血：
    - 明确区分三类修订：补证分支、分析重跑分支、纯表达重写分支。
    - 对 `rerunFromNode`、`resumeTask`、`resumeNode` 等命令补稳定回归测试。
    - 在回流节点配置中显式保留来源和原因摘要。
  - 治本：
    - 把 `CompensationGraphAssembler`、`DynamicTaskGraphService`、`TaskRuntimeCommandAppService` 组成独立的质量回流子域。
    - 正式建模“修订触发条件、受影响范围、回流目标链路、人工接管窗口”。
    - 建立修订与重写专题文档，避免该链路继续藏在运行时服务内部。
  - 提效：
    - 对受影响范围计算、节点回收、检查点重用引入缓存和批量更新。
    - 对重写类任务启用轻量执行模式，减少不必要的全链路回流。
    - 对高频重复回流动作设置预算闸门。

#### 2.1.8 对话协同

- 优化目标：
  让统一对话入口成为正式业务桥梁，而不是继续膨胀成“什么都往里塞”的超级服务。
- 核心技术栈 / 方案：
  - 模式识别契约
  - 动作预览协议
  - 确认执行协议
  - 页面上下文装配层
- 执行路径：
  - 止血：
    - 将解释、填表、动作预览、研究增强至少在响应结构上继续分清。
    - 明确高风险动作统一先走预览再确认，避免直接提交运行时命令。
    - 为对话链路补更多“页面上下文参与识别”的测试。
  - 治本：
    - 将 `ConversationService` 继续拆成会话管理、模式识别、动作桥接、检索增强四层。
    - 正式定义对话动作协议，与 `TaskRuntimeFacade` 之间只走正式命令对象。
    - 建立对话协同专题文档，把它和通用聊天彻底切开。
  - 提效：
    - 对常见解释和页面摘要做缓存。
    - 对上下文装配中的高频查询做批量化和延迟加载。
    - 对研究增强模式引入预算闸门，避免随手对话触发高成本检索链路。

#### 2.1.9 交付与审计

- 优化目标：
  让交付页优先服务“阅读结果、理解问题、执行下一步”，同时保证全过程审计可追溯。
- 核心技术栈 / 方案：
  - 报告交付投影
  - 审计摘要投影
  - 高级诊断后置层
  - 导出记录正式化
- 执行路径：
  - 止血：
    - 将交付页主路径收口到：交付状态、最大问题、建议动作、关键证据入口。
    - 把 `ReportService` 中仍混在一起的原始 JSON 兜底继续后置。
    - 为导出记录、审计摘要、证据入口补稳定契约测试。
  - 治本：
    - 拆分 `ReportService` 为交付主视图、证据视图、审计视图、导出视图。
    - 将事件回放和交付查询分离，不再由单一大服务聚合。
    - 建立交付与审计专题文档，定义主路径与高级诊断路径的边界。
  - 提效：
    - 对交付摘要、证据覆盖、审计摘要做缓存。
    - 对导出包生成和读取引入异步化。
    - 将高频读取的报告相关查询批量化。

### 2.2 业务执行引擎优化

#### 2.2.1 任务执行引擎

- 优化目标：
  让任务执行引擎只承接正式业务语义，而不是继续变成“所有运行时问题的大收纳箱”。
- 核心技术栈 / 方案：
  - `DagExecutor` 职责拆分
  - 节点状态机正式化
  - 共享上下文投影化
- 执行路径：
  - 止血：补状态机边界测试、限制共享上下文整包回灌、补事件缺字段。
  - 治本：把调度、执行、状态推进、事件发布、动态补图拆成协作对象。
  - 提效：批量状态更新、共享上下文裁剪缓存、异步事件投递优化。

#### 2.2.2 搜索执行引擎

- 优化目标：
  让搜索计划、验证、补源、正式选源、搜索回放成为独立子域。
- 核心技术栈 / 方案：
  - 搜索计划对象
  - 搜索现场对象
  - 搜索策略与候选选择器
- 执行路径：
  - 止血：直接吸收 `CollectorAgent.md` 的问题清单。
  - 治本：把搜索子域从 `CollectorAgent` 与外围大服务中拆出。
  - 提效：搜索缓存、快照复用、反爬熔断和预算门槛。

#### 2.2.3 质量回流引擎

- 优化目标：
  让质量问题到后续动作之间的转换不再散落在 Reviewer、命令服务、动态补图服务里。
- 核心技术栈 / 方案：
  - 回流动作协议
  - 受影响范围分析器
  - 动态分支模板
- 执行路径：
  - 止血：补齐动作分类和命令测试。
  - 治本：将回流动作和动态补图收口为独立引擎。
  - 提效：受影响范围缓存、重复回流抑制、轻量重写通道。

#### 2.2.4 对话动作引擎

- 优化目标：
  让对话动作预览、确认、正式执行成为受控桥梁，而不是对话服务内部字符串判断。
- 核心技术栈 / 方案：
  - 动作预览 DTO
  - 高风险动作确认协议
  - 正式命令桥接器
- 执行路径：
  - 止血：统一预览与确认响应形状。
  - 治本：把动作翻译和正式执行桥接独立出来。
  - 提效：常用解释缓存、上下文最小装配、成本门槛。

### 2.3 平台底座优化

#### 2.3.1 知识摄取 / RAG / 记忆

- 优化目标：
  让知识底座成为所有业务链路共享的正式事实层，而不是各链路各自拼接资料旁路。
- 核心技术栈 / 方案：
  - `KnowledgeDocument`、`RetrievalChunk`、`MemorySnapshot` 分层
  - 任务级 / 领域级 / 组织级隔离策略
- 执行路径：
  - 止血：补齐来源分类和回指元数据。
  - 治本：收口多级检索与多级记忆边界。
  - 提效：索引缓存、批量召回、热知识投影。

#### 2.3.2 AI 与工具治理

- 优化目标：
  让模型调用与工具调用都运行在统一治理边界内。
- 核心技术栈 / 方案：
  - `ModelGateway`
  - Provider 策略路由
  - Tool permission / timeout / fallback
- 执行路径：
  - 止血：补充异常、超时、重试与审计缺口。
  - 治本：业务层彻底去除对单一 Provider 细节的依赖。
  - 提效：成本路由、预算闸门、模型分层调度。

#### 2.3.3 可观测

- 优化目标：
  让任务、节点、搜索现场、Agent 输出、工具调用在同一观察体系里被解释。
- 核心技术栈 / 方案：
  - 事件流 DTO
  - 搜索现场观察对象
  - 统一日志与快照
- 执行路径：
  - 止血：统一关键事件字段。
  - 治本：拆清任务快照、节点事件、搜索现场三类视图。
  - 提效：事件投影缓存、增量推送优化。

#### 2.3.4 恢复与回放

- 优化目标：
  让恢复不仅能继续执行，还能解释“恢复前到底发生了什么”。
- 核心技术栈 / 方案：
  - 恢复检查点
  - 节点级回放帧
  - 搜索现场持久化回放
- 执行路径：
  - 止血：优先补齐搜索链路回放字段和检查点保留。
  - 治本：将回放从内存最近事件提升为正式持久化能力。
  - 提效：分层快照、增量回放、跨重启快速恢复。

#### 2.3.5 质量 / 配额 / 安全治理

- 优化目标：
  让质量、预算、安全成为全链路共享的治理面，而不是每条链路各自硬编码。
- 核心技术栈 / 方案：
  - 质量规则对象
  - 任务/搜索/模型预算闸门
  - URL 安全与来源可信度规则
- 执行路径：
  - 止血：把明显的校验与预算缺口补到统一入口。
  - 治本：将三类治理对象的规则和审计各自正式化。
  - 提效：风险预判、预算预占、异常聚合统计。

#### 2.3.6 多渠道资料接入

- 优化目标：
  让上传资料、认证源、未来连接器都进入统一资料接入语义，而不是分别挂靠在孤立接口上。
- 核心技术栈 / 方案：
  - 资料接入标准 DTO
  - 连接器定义与同步记录
  - 来源可信度与生命周期元数据
- 执行路径：
  - 止血：统一上传资料与认证资料的最小元数据。
  - 治本：将连接器入口纳入知识摄取主线。
  - 提效：增量同步、来源模板复用、权限缓存。

## 3. 跨链路约束层

| 共享契约 / 红线 | 统一规则 | 落地载体 |
| --- | --- | --- |
| Facade 互调规则 | Controller 只依赖 Facade / AppService；跨链路正式调用优先走 Facade，不直接越过边界直连底层服务或仓储 | `TaskRuntimeFacade`、`TaskQueryFacade`、`ReportQueryFacade`、未来各链路 Facade |
| Agent 持久化红线 | Agent 只负责结构化业务结果，不直接承担正式持久化职责；历史白名单只能收缩不能扩散 | `BackendModuleDependencyTest`、`ArchitectureWhitelist` |
| `sourceUrls` 可追溯红线 | 所有关键结构化对象、诊断对象、导出对象都必须可回指来源；缺少来源只能显式标记缺口，不能静默省略 | `EvidenceFragment`、`QualityDiagnosis`、`RevisionDirective`、`ReportResponse` |
| 预览态 / 运行态分离 | 计划预览、运行快照、节点详情、搜索现场不能共用“看起来一样但语义不同”的弱契约 | `TaskNodeResponse`、计划预览 DTO、运行态 DTO |
| 共享上下文裁剪规则 | 共享上下文只注入稳定业务事实，不回灌整包节点原始 JSON；恢复复用字段和调试字段必须分层 | `AgentContext`、`TaskSnapshotCacheService`、`DagExecutor` |
| Golden Master 命名规范 | 对抽取、分析、写作、质检、交付等结构化输出，统一使用 `*GoldenMasterTest` 或 `*SnapshotContractTest` 命名，避免快照测试随意散落 | 后续各链路专题测试资产 |
| ArchUnit 边界规则 | 新增专题时必须先决定边界规则，再决定白名单；白名单必须记录原因、回收阶段和 owner | `BackendModuleDependencyTest`、`ArchitectureWhitelist` |
| PR 合并顺序 | 统一按 `契约 -> 引擎 -> Agent / Service -> 查询投影 / UI` 顺序合并，避免先改页面再倒推底层语义 | 研发流程约束 |
| 专题文档命名规范 | 每条主业务链路都应最终形成一份独立专题文档，建议沿用 `docs/problem/<ChainName>Agent.md` 或等价统一命名 | 当前基线：`CollectorAgent.md` |

## 4. TDD/SDD 落地指导 (Execution Blueprint)

### 4.1 核心优化点 A：任务定义与编排

#### 核心业务故事 (Story)

作为任务创建者，我希望在任务启动前就看到一份与真实执行一致的“任务定义与编排预览”，而不是只看到一堆表单字段或一个脱离运行时的假计划。  
系统需要基于任务目标、研究对象、资料策略和输出要求，生成正式计划语义，并在后续运行、重跑、恢复时继续沿用这份语义。

#### 关键测试用例 (Test Cases)

```java
// 伪代码：任务定义与编排的 Red 阶段断言
class TaskDefinitionAndOrchestrationContractTest {

    @Test
    void should_expose_business_plan_semantics_in_preview_response() {
        // 红灯断言：预览响应必须带出阶段、分支、回退策略等正式计划语义，
        // 不能只有节点列表和原始配置 JSON。
    }

    @Test
    void should_keep_preview_contract_separate_from_runtime_node_contract() {
        // 红灯断言：预览态 DTO 与运行态 DTO 不得混用同一组弱字段，
        // 否则前端无法区分“计划值”和“真实执行值”。
    }

    @Test
    void should_preserve_plan_version_when_rerun_or_resume_task() {
        // 红灯断言：重跑或恢复任务时，系统必须能指出当前继续沿用的是哪一版计划，
        // 不允许只重置节点状态却丢失计划版本语义。
    }

    @Test
    void should_fail_fast_when_task_contract_misses_required_business_fields() {
        // 红灯断言：如果新版任务定义缺失关键业务字段或字段语义冲突，
        // 系统应在定义阶段失败，而不是等到运行期才暴露。
    }
}
```

### 4.2 核心优化点 B：搜索与采集

#### 核心业务故事 (Story)

作为研究任务系统，我希望在用户只提供少量线索时，能够自动完成高质量补源和正式采集。  
系统不仅要尽量找对来源，还要在动态网页、营销噪声、反爬阻断、搜索降级等复杂现实下，仍然保持结果可解释、可恢复、可回放。

#### 关键测试用例 (Test Cases)

```java
// 伪代码：搜索与采集链路的 Red 阶段断言
class SearchAndCollectionGoldenMasterTest {

    @Test
    void should_reject_official_marketing_page_even_when_domain_authority_is_high() {
        // 红灯断言：即使候选链接属于官方域名，只要命中营销噪声特征，
        // 也不能因为域名权威高就自动通过验证。
    }

    @Test
    void should_accept_high_value_chinese_pricing_document() {
        // 红灯断言：中文“价格/计费/收费/套餐”类页面应能被正确召回和验证，
        // 不能因为词库只有英文而被误杀。
    }

    @Test
    void should_reuse_verified_collected_page_instead_of_fetching_same_url_twice() {
        // 红灯断言：验证阶段已经抓到的网页快照，
        // 正式采集阶段必须优先复用，而不是再次发起相同网络请求。
    }

    @Test
    void should_publish_search_audit_and_selected_targets_to_replay_stream() {
        // 红灯断言：搜索链路完成后，回放和实时事件至少要能重建
        // searchAudit、selectedTargets、degradationReason 等关键现场。
    }

    @Test
    void should_keep_search_checkpoint_when_rerun_from_collector_node() {
        // 红灯断言：从采集节点重跑时，系统必须保留可复用的搜索检查点，
        // 不能把高价值搜索现场一刀切清空。
    }
}
```

### 4.3 SDD 落地顺序建议

1. 先写母图级设计，再写链路专题，不反过来。
2. 先冻结正式业务契约，再拆大类实现，不反过来。
3. 先为“任务定义与编排”和“搜索与采集”建立可执行测试骨架，再推广到抽取、分析、写作、质检。

## 5. 最终建议

1. 下一阶段最先应推进的不是“继续补某个 Agent 的细节”，而是把 `任务定义与编排` 正式提升为第 1 条主业务链路。
2. `搜索与采集` 是当前最成熟的深度样板，应以 `[CollectorAgent.md](/e:/java_study/Mul-agnet/docs/problem/CollectorAgent.md)` 为基线整理成三段式路线，而不是再另起炉灶。
3. 从当前时点开始，后续每补一条专题文档，都应先标明：
   - 它属于哪条主业务链路
   - 由哪个执行引擎承接
   - 触碰了哪些平台底座
   - 受哪些跨链路红线约束
4. 当 `Extractor / Analyzer / Writer / Reviewer / Conversation` 几条专题逐步补齐后，本文档将成为它们的总控目录，而不是一次性的路线图。
