# 2026-06-12 会话总结

## 核心结论

**当前 specs 的 2.1.2 搜索与采集计划是"参数止血 + 契约冻结"，不是真正的架构优化。** 它修了 OFFICIAL 免检、补了中文词库、收了 fallbackOrder 入口，但没有触及 CollectorAgent.md 暴露的架构级问题（Provider 无脑全量串扫、预览/执行两套世界、HEURISTIC 伪能力、公网广告引擎默认主力等）。

**真正的问题根因：没有诊断就写不出优化方案。** CollectorAgent.md 是唯一完成深度诊断的链路，所以只有它的方案是扎到肉里的。其余 8 条链路的"优化方案"是同一个模板（拆类 + 建文档 + 定契约），因为没有逐行读代码就无法写出有针对性的东西。

## 已确认的链路推进闸门

硬规则，不可跳步：

1. 先写诊断（docs/problem/<ChainName>.md）
2. 诊断确认 blocking 项
3. 再写方案（docs/superpowers/plans/*.md）
4. 再写实施计划
5. 最后做代码与验证
6. 回链到 specs 更新状态

**没有诊断，不写方案。没有方案，不开大实施。没有验证，不标完成。**

## 状态列定义

| 列 | 含义 | 通过条件 |
|----|------|----------|
| 诊断 | 已有问题证据、边界判断、blocking 项归类 | 诊断文档存在且覆盖链路主路径 |
| 方案 | 已形成被确认的目标边界和改造思路 | 计划文档存在，明确写出做什么、不做什么与验收口径 |
| 实施 | 代码已落地，自动化测试通过，必要文档已更新 | 关联实现已落地，单元/集成/架构测试通过 |
| 实链验证 | 真实场景端到端验收 | 同时满足 4 条硬条件 |

### 实链验证 4 个硬条件

1. 真实场景 — 跑的是代表性真实链路，不是只跑 mock
2. 产出正确 — 主路径产出与预期一致，不只是"没报错"
3. 过程可观测 — 关键中间状态完整可见
4. 恢复/回放/重跑 — 涉及这些行为的链路，必须验证至少一次

### 含义硬约束

- 自动化测试归**实施**，不归**验证**
- 验证 = 真实运行，不是自动化测试的别名

## 最新 specs 文档评估

新版 specs 已重写，删除了所有未经诊断的模板内容。评估结果：

- **方向正确**：地图归地图、诊断归诊断、方案归方案
- **删干净了**：止血/治本/提效模板全部移除
- **状态诚实**：没有链路被错误标成"已完成"

### 待修复项（按优先级）

1. **引擎和底座的状态追踪完全缺失。** 新版只有 9 条链路的状态看板。4 个引擎和 6 个底座在 1.3/1.4 只列了叙述，没有成熟度追踪。引擎建议用"隐式存在 → 已识别 → 独立子域"三级成熟度模型；底座用"未建立 → 已有基础缺入口 → 统一入口"三级。不要粘回旧的"止血/治本/提效"模板。

2. **1.2 未诊断链路的"业务痛点"列**需要标注为 `[未诊断—推测]` 或改为 `⬜ 待诊断`。当前文字是从旧版继承的通用叙述，没有逐行代码阅读支撑。

3. **Section 3 的 3.3~3.9** 是 7 份相同模板的重复。建议缩为一张索引表，只保留 3.1（任务定义与编排）和 3.2（搜索与采集）为展开的链路详情。

4. **🟡 状态缺少明确的升 ✅ 条件。** 每条标 🟡 的链路/引擎/底座，应该在下面写一句硬话：完成什么具体条件后可以标 ✅。

5. **框架本身的微调：**
   - 质量回流引擎和对话动作引擎标注为"准引擎"（桥接语义存在，待独立）
   - 知识摄取/RAG/记忆 当前合并可接受，但三个概念（怎么进/怎么查/怎么沉淀）后续可能需要拆
   - 质量/配额/安全治理 的"质量"和链路"质量审查"、引擎"质量回流"三重撞名，需要区分命名
