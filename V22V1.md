# V2 相对 V1 的实际升级说明

> 对比基线：`projectV1.md`
> 说明原则：只记录当前工程里已经真实落地的能力，以及还未完成的部分。

---

## 1. 当前判断

当前工程不是“V2 全部完成”，也不是“只比 V1 多一点点”。

更准确的定位是：

**当前版本已经从 V1 的可演示 MVP，升级到一个具备节点级溯源、自动质检闭环、第一版动态 DAG、第一版补源、第一版断点续跑的 V2 中期版本。**

也就是说，前面做的代码并不是没价值，而是已经把 V2 最核心的主链路先搭起来了；现在剩下的，更多是“第一版继续做深做全”，不是完全空白。

---

## 2. V2 任务状态总览

为了避免“看起来好像还剩很多，但不知道前面到底完成了什么”，这里按三类来拆：

1. 已完成
2. 已做第一版，还需增强
3. 基本未做

---

## 3. 已完成

这些部分可以认为已经真正落地，不只是停留在计划里。

### 3.1 节点级溯源面板

已完成内容：

1. TaskDetailPage 已支持 `Node Trace`
2. 可查看节点 `config / input / output`
3. 可查看上游依赖 `Upstream Nodes`
4. 可查看 `Evidence Trace / Evidence Captured`
5. Reviewer 节点可展示评分、是否通过、问题列表、修订计划

这部分已经不再是 V1 那种“只看 DAG 状态”，而是能解释节点为什么这么跑。

### 3.2 Writer / Reviewer 自动闭环

已完成内容：

1. `quality_check`
2. `rewrite_report`
3. `quality_check_final`
4. Reviewer 失败后生成 `revisionPlan`
5. Writer 根据 `revisionPlan` 重写
6. 最终任务成功与否纳入终审结果判断

这部分已经实现了“初稿 -> 质检 -> 修订 -> 终审”的主闭环。

### 3.3 报告页闭环视图

已完成内容：

1. `Initial Review`
2. `Revision Plan`
3. `Rewrite applied / No rewrite needed`
4. `Final Review`
5. `Latest Quality Feedback`

这意味着报告页已经不是单纯的 Markdown 预览页，而是报告生产过程页。

### 3.4 报告接口闭环契约

已完成内容：

1. `initialReview`
2. `revisionPlan`
3. `rewriteApplied`
4. `finalReview`

这部分说明闭环能力已经从后端执行层延伸到 API 契约层。

---

## 4. 已做第一版，还需增强

这些部分不是没做，而是已经有第一版实现，但距离 V2 计划里的“完整版”还有差距。

### 4.1 动态 DAG 规划

当前已做：

1. 采集节点从固定单节点，升级为按“竞品 + sourcePlan”动态展开
2. 每个采集节点携带 `sourceType / sourceScope / discoveryNotes / schemaName`
3. 任务预览和节点追踪里能看到动态节点配置

还缺：

1. 按任务类型、模板、Schema 自动规划完整 DAG
2. 更细粒度条件分支
3. 人工调整 DAG 后再执行
4. 更强的计划持久化与可视化

当前状态结论：

**已做第一版，不再是固定 DAG，但还不是最终意义上的动态规划 DAG。**

### 4.2 自动信息源发现

当前已做：

1. 新增 `SourceDiscoveryService`
2. 新增 `HeuristicSourceDiscoveryService`
3. 支持基于用户 URL 自动扩展 docs / pricing / news / review
4. 支持基于竞品名推断根域名
5. REVIEW 类型补充公开测评入口

还缺：

1. 真实搜索式补源
2. 去重
3. 来源评分
4. 时效排序
5. 可信度判断

当前状态结论：

**已做第一版启发式补源，但还不是完整的自动信息源发现系统。**

### 4.3 字段级证据约束

当前已做：

1. Extractor 提示词明确要求 `sourceUrls`
2. `SchemaExtractorAgent` 会补齐 `sources / sourceUrls / evidenceCoverage`
3. `CompetitorKnowledge` 已持久化 `sourceUrls / evidenceCoverage`
4. 对 `summary / positioning / targetUsers / coreFeatures / pricing / strengths / weaknesses` 做字段级状态标记

还缺：

1. 报告章节级证据约束
2. 结论级证据约束
3. Reviewer 对缺证据字段做硬校验
4. Writer 重写时强制修复缺证据字段
5. 前端更直观看到字段级缺证据告警

当前状态结论：

**已做第一版字段级溯源摘要，但还没有变成完整的强约束体系。**

### 4.4 导出能力

当前已做：

1. Markdown 导出
2. HTML 导出
3. 前端新增 `Export HTML`
4. HTML 中包含报告正文、质量状态、闭环结果、竞品溯源摘要、证据表格

还缺：

1. PDF
2. Word
3. PPT
4. 更正式的导出模板与版式

当前状态结论：

**已从单 Markdown 升级到 Markdown + HTML，但离完整交付导出能力还有差距。**

### 4.5 节点级恢复与人工干预

当前已做：

1. `resumeTask`
2. 保留成功节点输出
3. 只重置失败 / 跳过 / 中断节点
4. 执行前把成功节点输出重新注入 `AgentContext.sharedState`
5. 前端新增 `Resume`

还缺：

1. 指定节点重跑
2. 从某个节点重新编排下游
3. 人工修改节点配置后继续执行
4. 人工审批 / 驳回后继续流转
5. 节点级暂停 / 跳过 / 强制终止

当前状态结论：

**已做第一版断点续跑，但还不是完整的节点级人工干预编排。**

---

## 5. 基本未做

这些不是“第一版没做深”，而是还没有真正进入工程主体。

### 5.1 真实搜索式补源

当前缺：

1. 搜索引擎接入
2. 搜索结果筛选
3. 结果去重聚合
4. 来源质量评分
5. 时效性排序

### 5.2 报告章节级 / 结论级证据硬校验

当前缺：

1. 章节是否必须带证据的规则
2. 结论是否必须能追溯到字段证据的规则
3. Reviewer 对证据不足问题的硬拦截
4. Writer 自动修复缺证据章节

### 5.3 正式交付导出格式

当前缺：

1. PDF
2. Word
3. PPT
4. 附录、目录、封面、分页等正式版式

### 5.4 指定节点重跑与人工重编排

当前缺：

1. 从指定节点手动重跑
2. 对下游节点做选择性失效
3. 人工改配置后继续
4. 节点级审批流

---

## 6. 现在还剩什么任务

如果只看“还剩什么”，其实主要是下面这些：

### 6.1 高优先级

1. 指定节点重跑与人工干预编排
2. 真实搜索式自动补源
3. 报告章节级证据强校验

### 6.2 中优先级

1. 更完整的动态 DAG 自动规划
2. 更强的字段级证据修复闭环
3. PDF / Word 导出

### 6.3 低优先级

1. PPT 导出
2. 更复杂的多模板编排
3. 更高级的任务计划可视化

---

## 7. 最后结论

所以现在不能说“V2 还剩这么多，之前代码是不是没干活”。

更准确的说法是：

1. **V2 最关键的主链路已经搭起来了**
2. **剩下的任务主要是把几个第一版能力继续做深做全**
3. **真正还接近空白的，主要是搜索式补源、章节级证据硬校验、正式交付导出、指定节点重跑**

如果后面继续推进 V2，最合理的顺序还是：

1. 指定节点重跑与人工干预编排
2. 真实搜索式自动补源
3. 报告章节级证据强校验
4. PDF / Word 导出
