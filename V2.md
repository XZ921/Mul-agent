# AI 驱动的竞品分析 Agent 协作系统 V2 计划

## 1. V1 现状总结

V1 已经完成了一个可以演示的端到端主链路，并且我本地验证了 `backend` 可编译、`frontend` 可构建。

### 已完成

- 已完成前后端分离，后端提供任务、报表、Schema、Agent 日志等 REST API，前端提供任务列表、新建任务、任务详情、报表预览四个核心页面。
- 已完成固定 5 节点 DAG 流程，顺序为 `collect_sources -> extract_schema -> analyze_competitors -> write_report -> quality_check`。
- 已完成 5 个 Agent 角色的基础实现，分别是采集、抽取、分析、撰写、质检。
- 已完成公开信息采集的双模式策略，轻量 HTTP 抓取优先，不足时回退到 Playwright 浏览器渲染。
- 已完成竞品知识、证据源、报表、Agent 执行日志等数据落库。
- 已完成 Markdown 报告生成、页面预览和 Markdown 文件导出。
- 已完成 Agent 日志可视化，能看到输入、输出、prompt、token、耗时、traceId。
- 已完成基础 Schema 管理页面和接口。

### 部分完成

- 有质量检查，但目前只是单次审查，未形成“审查失败 -> 反馈修改 -> 重新生成”的闭环。
- 有证据链和引用展示，但还没有做到字段级、结论级、章节级的强约束溯源。
- 有 `retryable`、`maxRetries` 等字段，但 DAG 执行器还没有真正做节点级重试和断点续跑。
- 有 Schema 管理，但还没有把 Schema 真正变成任务规划和 DAG 生成的核心输入。

### 仍然缺失

- 没有动态 DAG 生成，当前是固定五节点流程，无法按任务类型、分析维度、竞品数量自动变更流程。
- 没有自动发现信息源，当前更依赖用户提供 URL，缺少搜索、去重、排序和补源能力。
- 没有多轮交叉审查和反馈闭环，质检结果不会自动驱动重新抽取或重写。
- 没有更强的证据结构化，缺少字段级证据引用、原文片段定位、发布时间等精确元数据。
- 没有 PDF、Word、PPT 等更完整的报表导出能力，目前优先是 Markdown。
- 可观测性主要停留在日志和数据库记录，还没有成本统计、链路追踪面板、质量分析看板。
- 默认运行仍依赖数据库和 AI 提供方配置，真正的开箱演示还需要进一步收敛 mock / demo 配置。

## 2. V2 的目标

V2 的目标不是简单“把功能做多”，而是把系统从一个可演示的 MVP 升级成一个真正可用于企业竞品研究的协作式 Agent 系统。

V2 重点解决 4 件事：

- 让任务规划从“固定流程”升级为“基于 Schema 的动态 DAG”。
- 让信息分析从“单次生成”升级为“证据驱动的结构化分析”。
- 让质检从“打分”升级为“能触发修订的闭环”。
- 让系统从“能看结果”升级为“能完整追溯每一步为什么这么做”。

## 3. V2 需要补齐的核心能力

### 3.1 动态任务规划与 DAG 生成

- 根据任务主题、竞品数量、分析维度、信息源范围、报表模板自动生成 DAG。
- 支持按竞品并行采集、按维度并行抽取、按章节并行撰写。
- 让节点定义真正使用 `dependsOn`、`required`、`retryable`、`maxRetries`、`executionOrder` 等字段。
- 支持节点级重试、失败恢复和断点续跑，而不是只能整任务重跑。
- 让 DAG 既能固定演示，也能根据任务规模自动扩展。

### 3.2 信息源发现与采集升级

- 增加自动发现信息源能力，优先补齐官网、文档、定价页、博客、新闻、公开评测等来源。
- 支持搜索驱动的采集策略，而不是只依赖用户手动填写 URL。
- 增加来源去重、相关性排序、发布时间排序和置信度排序。
- 让采集结果保存更完整的元数据，包括标题、URL、发布时间、采集时间、原文片段、页面类型、来源质量评分。
- 继续保留 HTTP 抓取与浏览器渲染回退机制，但要把采集策略参数化。

### 3.3 Schema 与中间契约升级

- 把竞品知识 Schema 从“存一个 JSON”升级为“字段级、证据级、可复用的强类型契约”。
- 建议继续拆分并强化这些契约对象：
  - `CollectResult`
  - `CollectedDocument`
  - `ExtractResult`
  - `CompetitorKnowledgeDraft`
  - `AnalysisResult`
  - `ReportDraft`
  - `QualityCheckResult`
  - `RevisionPlan`
- 每个关键字段都尽量绑定至少一个证据引用，避免结论悬空。
- 在 Schema 里增加置信度、证据数、来源质量、结论类型等辅助字段，方便后续质检和人审。

### 3.4 分析、撰写与质检闭环

- 分析 Agent 输出结构化横向对比，而不是只输出一段大段文本。
- 撰写 Agent 基于结构化分析结果生成 Markdown 报告，并按章节组织。
- 质检 Agent 不只打分，还要输出具体问题列表、问题等级、建议修订方案。
- 增加交叉审查 Agent 或事实核查 Agent，专门检查关键结论是否被证据支持。
- 当质检不通过时，系统应能自动生成修订计划，并把问题反馈给撰写 Agent 或抽取 Agent 重新运行。
- 保留人工确认入口，让用户可以决定是否接受修订建议并重新执行。

### 3.5 报告与导出能力

- 报告正文必须有清晰章节结构，摘要、竞品概览、功能对比、定价对比、用户画像、差异化分析、机会与风险、建议、证据列表都要有。
- 报告中的关键结论必须带证据编号，证据列表必须可回溯到原始 URL 和原文片段。
- 支持 Markdown 导出作为第一优先级。
- V2 应补充 PDF 导出，后续再考虑 Word 和 PPT。
- 支持报告模板切换，例如标准版、精简版、管理层版。
- 支持章节跳转和证据展开查看。

### 3.6 可观测性与审计

- 继续保留 `traceId` 贯穿 API、任务、节点、Agent 和日志。
- 每个 Agent 必须记录输入、输出、模型名称、token 用量、耗时、prompt、错误原因、人工介入标记。
- DAG 节点需要记录开始时间、结束时间、状态变化、失败原因和重试次数。
- 增加质量问题历史、修订历史和执行链路回放能力。
- 后续如果条件允许，可以接入成本统计和链路观测面板。

### 3.7 前端体验升级

- 新建任务页需要支持 Schema 选择、信息源范围选择、报表模板选择、预览 DAG。
- 任务详情页需要支持节点级状态、重试按钮、日志详情、产物预览和修订反馈。
- 报告页需要支持证据列表、章节折叠、引用展开、质检问题展开。
- 增加“重新执行”“只重跑失败节点”“接受修订并重试”等操作入口。
- 保留当前简洁的演示路径，但让它更像一个真实的研究工作台。

## 4. V2 推荐实施顺序

### 第一阶段：把执行器做成真正的 DAG 引擎

- 完成动态 DAG 的创建与持久化。
- 完成节点依赖校验、节点级状态流转、节点级重试。
- 完成任务重启后按节点状态恢复，而不是整任务盲重跑。

### 第二阶段：把 Schema 变成真正的规划输入

- 完善 Schema 数据结构与契约对象。
- 让任务创建时可以根据 Schema 生成 DAG。
- 让抽取结果、分析结果、报告结果都能回写到结构化对象里。

### 第三阶段：把信息源发现与证据链补强

- 增加自动发现信息源能力。
- 增强采集结果的元数据和证据定位。
- 让每条结论都尽量能回指到证据源。

### 第四阶段：把质检做成闭环

- 让质检结果真正影响下一轮撰写或抽取。
- 引入修订计划、重新执行和人工确认。
- 让报告的质量从“看起来完整”升级为“可验证、可修订”。

### 第五阶段：补齐导出、可观测性和演示体验

- 增加 PDF 导出。
- 补齐链路、日志、token、耗时、质量问题的展示。
- 打磨前端交互，让 demo 更稳定、更像产品。

## 5. V2 验收标准

- 用户可以通过 Schema 和任务输入自动生成可执行 DAG。
- 系统可以采集、抽取、分析、撰写、质检并形成闭环。
- 每条关键结论都能回溯到明确证据。
- 任务失败后可以按节点恢复或按失败节点重试。
- 报告页可以清楚看到证据、质检结果和 Agent 执行过程。
- Markdown 导出可用，PDF 导出作为 V2 的增强目标。
- 后端和前端都能稳定构建，基础演示流程可重复跑通。

## 6. 暂缓项

以下内容不建议在 V2 里一次性做太重，可以留给后续版本：

- 用户登录、权限系统、团队空间。
- 私有数据源接入和企业知识库深度集成。
- 大规模并行调度与分布式任务编排。
- 全量 BI 仪表盘和成本分析平台。
- PPT 自动生成和复杂图表自动编排。

## 7. 结论

V1 已经证明了“多 Agent 协作生成可溯源竞品报告”这条主链路是成立的。
V2 的重点不是重新发明系统，而是把这条主链路做成真正可配置、可恢复、可审查、可追溯的研究工作流。
